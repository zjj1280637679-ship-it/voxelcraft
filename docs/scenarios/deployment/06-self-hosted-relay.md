# 场景 06:自建公网中继(VPS 长期开服)

## 目标

把中继部署到一台有公网地址的服务器上,获得**固定、随时可连**的入口:
任何人打开 `http://你的IP:端口` 即可建房/进房,App 玩家把它设为服务器地址即可。
相比场景 02 的隧道:地址不变、延迟可控、不依赖你的家用电脑开机。

> ## ⚠️ 安全警告:先读这段再决定是否长期公开运营
>
> 当前版本的中继是为"朋友间开黑"设计的,**B 类安全加固尚未完成,
> 不建议在加固完成前把它作为长期、公开、匿名可达的服务运营**。
> 细节与加固清单见[白皮书](../../whitepaper.md)第 8 章"信任与安全模型"
> (尤其 8.2 节"B 类待补清单——公网长期开放前必须补")。
> 直接从代码可以确认的现状:
>
> - **无任何鉴权**:任何能连上端口的人都可建房、进房、拉取公开房间列表;
> - **无房间密码**:房间名是唯一门槛,私密房只是"不上列表",名字被猜到即可进入;
> - **无速率/数量限制**:除单帧 64 KB 上限与每房 8 人外,
>   没有连接数、建房数、消息频率的限制,易被滥用刷资源;
> - **明文传输**:裸 IP+端口走 `ws://`,内容(含房间名、玩家名、聊天性质的数据)可被途中窥探;
> - 中继不解析游戏数据(`d` 字段不透明转发),所以它不会"被游戏逻辑攻击",
>   但**恶意房主/成员对房间内其他玩家的体验破坏无从约束**。
>
> 建议:仅在可信小圈子里分发地址;长期运营前关注白皮书安全章的加固进展;
> 出问题随时 `Ctrl+C`——中继无状态,重启零成本(代价是当时的房间全部断开)。

## 前提

- 一台有公网 IP 的服务器(任意便宜 VPS / 轻量云均可),
  装好 Node.js(架构契约按 Node.js 24 编写,见 `DESIGN.md`)。
- 能 SSH 登录并配置防火墙/安全组。
- **备案说明**:以**裸 IP + 端口**(如 `http://1.2.3.4:8080`)提供服务不涉及
  域名解析,无需 ICP 备案;一旦绑域名对外服务,按所在地法规可能需要备案,请自行确认。

## 操作步骤

1. 上传代码并安装依赖(运行时只需要 `ws` 一个外部包):

   ```bash
   git clone <你的仓库地址> voxelcraft
   cd voxelcraft
   npm install --omit=dev    # 或直接 npm install
   ```

2. 启动。端口由环境变量 `PORT` 控制,不设则默认 8080,监听 `0.0.0.0`:

   ```bash
   PORT=8080 node server/index.js
   ```

   启动横幅会列出 `http://localhost:8080` 与各网卡地址
   (云服务器上列出的多是内网 IP,对外请用云控制台显示的**公网 IP**)。
3. 放行端口:
   - 云厂商**安全组**:放行 TCP 入方向 8080(或你设的端口);
   - 系统防火墙(如 ufw):`sudo ufw allow 8080/tcp`。
4. 保持常驻(可选但推荐)。最简单的 systemd 例子:

   ```ini
   # /etc/systemd/system/voxelcraft.service
   [Unit]
   Description=VoxelCraft relay
   After=network.target

   [Service]
   WorkingDirectory=/opt/voxelcraft
   Environment=PORT=8080
   ExecStart=/usr/bin/node server/index.js
   Restart=always

   [Install]
   WantedBy=multi-user.target
   ```

   `systemctl enable --now voxelcraft` 启用。(也可用 pm2 等进程管理器,仓库不绑定。)
5. 验证:
   - 本机浏览器开 `http://公网IP:8080` 能见到大厅且状态行显示已连接;
   - 手机流量下同样可达;
   - App 玩家在服务器地址栏填 `公网IP:8080`(裸 host:port → `ws://` 明文,
     这是预期行为,见场景 03 的地址规则表)。
6. (可选)给中继打包一份对应默认地址的 APK 分发:见场景 05 的 `-DefaultServer`。

## 预期结果

- 地址长期不变,玩家"我的足迹"历史房间、保存的服务器地址持续有效。
- 中继本身无游戏状态(没有种子、没有修改记录、没有坐标),内存占用与流量都很小;
  房间世界依然活在房主玩家的浏览器里,服从场景 04 的迁移规则。
- 服务器日志可见每个房间的创建/删除、成员进出与房主迁移。

## 常见故障与排查

| 现象 | 原因与处理 |
|---|---|
| 本机 curl 通,外网连不上 | 十有八九是云安全组没放行端口——它和系统防火墙是**两道闸**,都要开。 |
| 报"端口 xxxx 已被占用 (port in use)" | 已有实例在跑(或 systemd 已拉起一份);服务器会打印换端口提示。用 `PORT=别的端口` 或先停旧实例。 |
| 想用 80/443 端口 | Linux 上 1024 以下端口需特权;更稳妥的是用 Nginx/Caddy 反代到 8080 并顺手上 TLS(注意反代必须支持 WebSocket 升级,路径 `/ws`)。配好证书+域名后,玩家地址栏填裸域名即自动走 `wss://` 加密。 |
| 玩家说"已连接"但列表里看不到别人的房 | 房间列表只含公开房,且上限显示 50 条、新房在前;再确认双方填的是同一个地址(状态行会显示当前服务器)。 |
| 玩家频繁被踢 | 中继每 10 秒 ping 一轮,未回 pong 即断开;多见于玩家网络劣化,中继侧无须处理。 |
| 服务器重启后房间没了 | 设计如此:中继无状态,房间登记表在内存里;世界数据在玩家端,但中继一断玩家也会全部回大厅,需要重新建房。 |
| 怕被刷 | 见顶部安全警告——加固完成前别公开张贴地址;必要时改非常规端口、用安全组限源 IP 段。 |
