# 场景 04:房主退出与世界存续(房主迁移)

## 目标

理解并验证:房主中途退出或掉线时,房间**不会消失**——
中继自动把"最老成员"(最早加入且仍在线者)提升为新房主,世界无缝继续。
只有当无人可继承(房里只剩房主一人)时房间才会删除。

## 原理速览(为什么世界丢不了)

- v2 架构里**每个客户端都持有全量世界**:世界 = 地形种子 + 全部方块修改记录,
  网络上只传修改和位置。任何成员随时具备"接任房主"的全部数据,迁移因此廉价。
- 中继(`server/index.js`)按加入顺序记录成员;房主连接关闭时:
  - 还有成员 → 提升最老成员:给它发 `promote`(附房间名与剩余成员 id 列表),
    给其他成员广播 `newhost`;房间名不变,成员连接全程不断。
  - 没有成员 → 删除房间。
- 新房主(`public/js/main.js` + `public/js/host.js`)用本地世界和已知玩家信息
  重建权威逻辑,随后向全房广播 `resync`(自己的完整修改日志),
  让所有幸存成员收敛到新房主的权威世界。

## 前提

- 任一已跑通的联机场景(01/02/03 均可),房间内至少 2 人(房主 + 1 名成员)。

## 操作步骤(实测一次迁移)

1. A 创建房间,B、C 先后加入(B 是"最老成员")。三人各放几个方块,确认同步正常。
2. 让 A 退出,两种方式都可以验证:
   - **主动退出**:A 点游戏内右上角"退出"按钮回大厅;
   - **模拟掉线**:直接关掉 A 的标签页/断网。中继每 10 秒一轮 ping/pong
     活性检查,未应答的连接会被强制断开——所以非正常掉线的检测最长约两个
     心跳周期(约 20 秒),期间房间短暂"无主"。
3. 观察 B 和 C 的画面与服务器日志。

## 预期结果

- **B(被提升者)**:弹出提示"你已接任房主";A 的人物消失;游戏不间断。
  B 的标签页从此承担权威逻辑(若 B 是触屏设备会尝试申请屏幕常亮)。
- **C(普通成员)**:弹出提示"A 离开,B 接任房主";经历一次短暂的
  `newhost` + `resync`——表现为 A 的人物消失、世界与新房主对齐,通常无感知卡顿。
  C 还会自动向新房主重发一次 `hello`(幂等),避免成为新房主不认识的"幽灵成员"。
- **服务器日志**:一行 `host migrated #旧id -> #新id`。
- 房间名、房间内其余玩家的 id 均保持不变;后续新玩家照常可进。
- 若 A 退出时房里没有其他人:日志显示房间删除,世界随之消失(当前版本无存档)。

## 边界情况(设计内行为,不是 bug)

| 情况 | 行为 |
|---|---|
| 房主"死亡窗口"内有成员放的方块 | 旧房主已收不到、新房主也没见过的修改,会被 `resync` 覆盖丢弃——**以新房主的世界为准**(这也是后续新进玩家看到的版本)。窗口很短,正常游玩几乎碰不到。 |
| 死亡窗口内有成员离开 | 它的离开通知随旧房主丢失;新房主接任时按中继给的存活名单修剪,并替它向全房补发离开广播。 |
| 被提升者自己还没完成进房(没拿到世界) | 它无法当房主,会主动断开,中继顺位提升下一名成员。 |
| 连续掉线(新房主又掉) | 同一套机制递归生效,直到无人可继承。 |
| 手机房主锁屏 | 标签页被系统冻结等同掉线,触发迁移;所以长期房主建议用电脑(见场景 03)。 |

## 常见故障与排查

| 现象 | 原因与处理 |
|---|---|
| 房主关页面后,成员等了十几秒才看到迁移提示 | 正常:非正常断线靠 10 秒间隔的心跳检测兜底,最长约 20 秒;主动点"退出"则几乎即时。 |
| 迁移后某成员的世界有一两个方块和别人不一样 | 理论上 `resync` 会拉平;若复现,记下双方 `window.__vc.world.edits` 差异提 issue。`resync` 只补"有记录的修改",不会删除多余条目以外的状态。 |
| 全员退出后想"找回"房间 | 找不回:世界只活在玩家的浏览器内存里,所有房主迁移链断尽即消失。当前版本未做存档(见 README"当前限制")。 |
| 想指定谁继承房主 | 不支持:继承顺位固定为加入顺序最老者(中继按 Map 插入序取第一个成员)。 |
