# 《永远的山根乡》造物与制造体系 — 设计文档 v1

> **范围**:沉淀"内容如何被创造、又如何被玩家制造"这一整套体系——多轮辩证后定案的模型。
> 与 [`原型显现-阶段1.md`](原型显现-阶段1.md)(引擎落地)、[`../../DESIGN.md`](../../DESIGN.md)(现有架构契约)平行;**本文只定模型与代码骨架,不写实现**。
> 命名:游戏名《永远的山根乡》,VoxelCraft 为英文代号。

---

## 0. 一句话

内容由「上帝模式」创作、沉淀进「世界定义集」,玩家在「游玩层」用 **工作台 + 图纸 + 材料** 把定义集里**已有**的东西制造出来。
**创作无限、制造受限;房主即上帝、玩家只消费。**

---

## 1. 总览:两层 + 单向管线

```
 代码层素材池 ──取用──▶ 上帝模式(三层权益,按角色分配) ──产出──▶ 世界定义集 ──界定可造范围──▶ 游玩层 ──▶ 世界中的实例
 (贴图/模型,                (L1 无限资源 / L2 设计新事物 /        (房主增删·受预算)        (工作台阶级
  随构建发布)                 L3 增减素材·定义)                                          × 图纸 × 材料,AND)
```

**关键性质 —— 单向解耦**:游玩层只读定义集、永不回写;创作层只写定义集、不碰玩家库存。两层通过「定义集」这**一个契约**解耦,可各自独立演化与测试(见 §7.3)。

---

## 2. 本质规则:原型—显现(回顾)

| 原型(锁定) | 显现(自由) |
|---|---|
| 阶级 / 用途 / 掉落池 / 生态位 / 采集门槛 / 韧性 / 战斗定位 / 制造规则 | 名字 / 贴图 / 模型 / 动作包 / 攻击方式 / 生态位选择 / 附性 |

**铁律:表象可无限变,本质不可变。** 钻石色木剑仍是木级;叫「神兽」的木级猎物不能掉钻石、不能有 Boss 属性。
→ 这是整个体系平衡的根:**逻辑只认原型字段,显现再花也改不了结算**。

## 3. 创作层:上帝模式(三层权益)

### 3.1 三层
- **L1 无限资源**:台子里材料无限。
- **L2 在方块规则内设计新事物**:用已有积木 + 素材池 + 动作包**重组**出新定义。
- **L3 增减素材 / 定义**:删除已有(含内置,如蜘蛛)、管理花名册(roster);素材的代码层加入也归此层授权范围。
  > **注**:素材的代码层加入是**开发者构建期动作**(§3.4、§6.1:发新版 = 全员更新),**非运行期台内能力**——游玩内的 L3 房主在运行期并不能真的往素材池里加一张新贴图(那需改源码 + 重构建 + 全员更新)。此处仅指该授权在概念上归 L3 范畴;**运行期游戏内加素材待 §7.1 AssetSource 上传 / AI 管线重启后才成立**。

### 3.2 权限分配(类似 QQ 群)
| 角色 | L1 | L2 | L3 |
|---|:--:|:--:|:--:|
| 房主(群主) | ✓ | ✓ | ✓ |
| 管理员(房主指定) | ✓ | ✓ | ✕ |
| 玩家 | ✕ | ✕ | ✕ |

### 3.3 不验平衡
房主/管理员造新东西**不过平衡校验**——即上帝,合理性自负。受影响的只有自愿进该房的人。

**界线(收窄措辞,避免误读)**:本体系**取消的是白皮书 §13.7 对【上帝模式创作是否过强】的强度/平衡预算闸门**(被"房主信任 + 房主责任"取代);但**原型锁定的白名单 / 钳窗校验仍然保留**——显现只能引用 `allow.*` 内的素材/动作包、`tune` 钳到预算窗。它在 [`原型显现-阶段1.md`](原型显现-阶段1.md) §2.5 的 `validateManifest` 里落地,且是 §2.6 成员侧反作弊的**承重原语**(成员复跑同一 `validateManifest` 拒签超白名单/超窗 manifest),**不在本节取消之列**。一句笼统的"不需要预算校验器"指的只是 §13.7 那道平衡闸门,不波及原型锁定的白名单/钳窗校验。

### 3.4 备选素材池(代码层)
贴图/模型在做出 **AI 自动处理接口之前,只能在代码层加入**,进入素材池,**随构建发布、版本化**。L2 只能从池里取;定义只存「第几号素材」的**引用**。

> **连带简化(重要)**:因为素材随构建发布、定义只存引用,自定义物退化成**轻量引用**——[`原型显现-阶段1.md`](原型显现-阶段1.md) 里那套自定义 PNG 传输 / 图集分片 / 内容审核方案**因此暂时搁置**。代价见 §6 第 1 条(版本强一致)。**重启条件 = 做游戏内上传 / AI 管线**(接入点见 §7.1 AssetSource)。

## 4. 世界定义集(创作产物)

### 4.1 内容
= 该世界当前生效的定义集合 = **内置定义 −(房主删除)+(房主/管理员新增)**。
每条定义 = 原型引用 + 显现(素材引用 / 动作包引用 / 生态位 / 附性)。

### 4.2 房主增删 / curation
房主用 L3 增删定义,塑造本世界的内容花名册(删蜘蛛、加自定义生物……)。

### 4.3 预算上限 与「卡死归房主」
定义集 + 世界增量受**硬上限**(沿用阶段1:可迁移增量 **~2800**、图集 **256 槽(其中运行期自定义 192 槽)**、图集 **1MB**)。id 分区(0..8 内置 / 9..63 静态造物 / 64..255 运行期自定义)以阶段1 §2.4 的 id 分区表为权威,本节只引用其总量,不另立口径。
是否撞墙(卡死)**由房主负责**,房主靠增删 curation 管理。**真实上限待实测,测后写入《房主须知》**(暂不强制做实时仪表盘)。

> **搁置期瓦片槽来源(与 §3.4 重启条件呼应)**:在自定义 PNG 传输方案搁置期内,图集 256 槽由**代码层随构建发布的素材**占用(内置 0..8 + 静态造物 9..63),运行期 P2P ws 分片字节通道处于**休眠态**(不传字节、定义只存引用)。192 个运行期自定义槽在搁置期不经 ws 通道填充;**重启条件 = 做游戏内上传 / AI 管线(见 §3.4)**,届时 ws 分片通道激活、运行期自定义槽才由字节填充。

## 5. 游玩层:制造的三道门

玩家造一件东西,需三者齐备(**AND**):
1. **工作台·材质阶级** = 基础设施(能在第几阶干活);靠**造 / 升级工作台**解锁(木台 → 石台 → …)。
2. **图纸(配方)** = 能做哪件具体东西;靠**做任务 / 探险**获得(对应白皮书「技能书 = 制造许可」)。
3. **材料** = 消耗真实材料;**材料阶级决定成品阶级**。

> **工作台加工规则(房主拍板,详见 [经济与阶级矩阵.md](经济与阶级矩阵.md) §4.2)**:一座台只能加工 **≤ 自身阶级的天然材料 + 图纸解锁的特定人造材料**(成品阶级可高于台阶级,如铁级台 + 高级图纸 → 钢 = 黄金级);**不能加工高于自身阶级的天然材料**。即:**人造材料看图纸(可越台阶级),天然材料看台阶级(不可越级)**。这条同时是「成品阶级能否高于开工台阶级」的通用答案。

三门齐备 → 实例化出「定义集里已有的」成品。**玩家不能造新东西。**

> **阶级压制无免疫(房主拍板)**:阶级差只**减伤 / 减速 / 减产,绝不归零** —— 战斗上木剑站撸钻石龙理论可杀(差≥4 仍 1%),采集上石镐挖铁慢但**保底掉到**。防跳级靠惩罚性低效而非硬墙。详见 [经济与阶级矩阵.md](经济与阶级矩阵.md) §2。

## 6. 四条硬约束(已拍板)

1. **版本不符不得进房**:进房握手先比对(素材池版本 + 定义集)哈希,不符**直接拒、无降级**。代价:加素材 = 发新版 = 全员更新。
2. **删除语义**:房主删某物(如蜘蛛)→ 已刷出的实例**瞬间死亡 + 不再刷新**。(自定义方块被删 → 已放置的回退占位/空气,具体待实现时定,见 §9。)
3. **卡死归房主**:无自动阻止,房主用 curation 管;真实上限实测后写《房主须知》。
4. **权限委托**:房主可设管理员(类似 QQ 群),管理员共享 L1+L2,**L3 房主独有**。

## 7. 代码骨架与扩展点 ★ 工程核心节(留后路 · 清晰 · 抽象 · 复用)

> **总原则:逻辑只依赖接口与注册表,内容一律是数据。** 加内容 = 加数据,不动逻辑;换行为 = 实现一个接口,不改调用方。

### 7.1 注册表(数据驱动)
- **PrototypeRegistry** `id → 原型`(只读规则记录)。战斗/采集/制造逻辑**只读**原型字段;新增显现永不触碰这些逻辑。
- **AssetPool** `id → 素材`(贴图/模型/音效),代码层注册、版本化。定义只持 id。
  → **`id = 内容哈希`(content-addressed,钉死语义)**:对齐 [`原型显现-阶段1.md`](原型显现-阶段1.md) §3.2(降采样 32×32 RGBA 的 SHA-256 前 16 字节),全房间天然去重、内容寻址分发(阶段1 §3.5)、迁移后按需拉取全部建立在此之上。
  → **未来游戏内上传 / AI 管线 = 在同一 id 接口背后多挂一个素材来源,消费方零改动**(这是给"上传"留的后路);**前提:任何后挂来源(上传/AI)产出的字节必须按同一哈希规则铸 id**——消费方零改动正源于 id 语义恒定,绝不可改用自增等其它铸 id 方式。
- **ActionPackRegistry** `packId → 动作包**数据**`(states / transitions / budget,packId = 下标)。**动作包不是脚本、不是可执行代码,而是纯数据**(对齐 [`原型显现-阶段1.md`](原型显现-阶段1.md) §2.5.1)。显现只引用 `packId`;内核侧暴露的是一组**写死在内核里的固定行为原语**(`prim` 纯函数集,见 §7.2 `Primitive`),动作包数据只能参数化组合这些原语,不能引入新原语。新增行为分两种:**可用现有原语表达 → 往 `ACTION_PACKS` 加一行数据(零波及)**;**需要新原语(罕见)→ 在内核加一个 `prim`(唯一会动逻辑处,需评审)**。
- **BlueprintRegistry / DefinitionSet**:见 §7.3。

### 7.2 接口草图(伪代码,只定形状)
```js
// 原型:不可变的规则记录
Prototype { id, category, tier, dropPool, niche, harvestGate, toughness, combatRole, craftRule }

// 显现/蓝图:引用原型 + 表象(全是引用与小数据)
Manifestation { id, protoId, name, assetRefs:[assetId], actionPackId, niche, affix }

// 素材来源:统一 id 接口 —— 未来 AI/上传来源也实现它,消费方不变
// ★ id 语义恒定:id = 内容哈希(content-addressed),见下方说明
interface AssetSource { has(id); get(id); version(); }

// 行为包 = 纯数据(states/transitions/budget,packId=下标)。不是脚本、不是代码模块。
ActionPack { packId, states:[{id,name,prim,...params}], transitions:[{from,to,when,...}], budget }
// 内核侧的固定行为原语集合(纯函数,写死在内核里;新原语=动逻辑,需评审)
interface Primitive { step(ent, view, dt, emit); }   // wander / flee / chase / idle / ...
// 内核按 state.prim 派发到对应 Primitive;动作包数据只能挑/调这些原语,不能新增

// 制造门:可组合谓词 —— 加一条约束 = 加一个谓词,不改制造主流程
type Gate = (ctx) => boolean       // hasWorkbenchTier / hasBlueprint / hasMaterials
craftable(def, ctx) = GATES.every(g => g(def, ctx))

// 权限 = 数据驱动的能力集(角色→能力列表是数据表) —— 加权益(未来 L4)= 加一个 capId
Privilege { caps: Set<capId> }     // 判定 = priv.has(cap)
priv.has(cap)                      // 房主含全部 cap / 管理员含 L1+L2 cap / 玩家空集
```

### 7.3 解耦缝(producer → consumer)
唯一**可变层**契约 = **DefinitionSet**。创作层(上帝)**只写**它;游玩层 / 模拟内核 / 渲染**只读**它。两侧独立演化、独立测试。

解耦实为**两层 + 两道哈希闸门**(§6.1 握手 = 比对这两个哈希):
- **(a) 代码层注册表**(PrototypeRegistry / AssetPool / ActionPackRegistry):随客户端版本发布、不可运行期更改,由 **`REGISTRY_HASH`** 兜底(对齐 [`原型显现-阶段1.md`](原型显现-阶段1.md) §2.7)——不匹配 = **连不进房**,这是防房主篡改注册表的核心闸门。
- **(b) 每世界可变的 DefinitionSet**:只持代码层注册表的 **id 引用**,自带 **`contentHash`**(对齐 §6.1)锁本世界可变层。

即 §7.1 列的四个注册表里,前三个(原型/动作包/素材)属代码层、由 `REGISTRY_HASH` 覆盖;DefinitionSet 是可变层、由 `contentHash` 锁。**落地者需校两个哈希,不是一个。**

### 7.4 与双系统 / 现有引擎的衔接
- 行为包是纯数据,内核的行为原语(`prim`)是纯函数(无 DOM/定时器)→ **直接复用 [`原型显现-阶段1.md`](原型显现-阶段1.md) 的纯模拟内核**,浏览器 rAF 与服务器 tick 两处共跑同一份。
- DefinitionSet / AssetPool 引用进世界状态与迁移快照,受 §4.3 预算。
- 轻量模式:创作权威 = 房主(本地信任);重型模式:权威 = 服务器。**同一套注册表/接口,只换权威与持久化层。**

### 7.5 留后路清单(明确的扩展位)
| 想加什么 | 怎么加 |
|---|---|
| 新内容(方块/生物/物品) | 往注册表加数据(零波及) |
| 新行为(可用现有原语表达) | 往 `ACTION_PACKS` 加一行数据(零波及,对齐阶段1 §2.5) |
| 新行为(需新原语,罕见) | 在内核加一个 `prim`(**唯一会动逻辑处,需评审**;对齐阶段1 §2.5/§2.7) |
| 新权益(未来 L4) | 能力表加一个 `capId` + 房主能力列表加一项(判定点零改动) |
| 新制造约束 | 加一个 Gate 谓词(零波及) |
| 游戏内上传 / AI 管线 | 实现 AssetSource 挂到 AssetPool 背后(产出字节须按内容哈希铸 id,§7.1) |
| 重型模式 | 替换权威/持久化层,接口不变 |

> **实施策略**:先搭骨架与桩(空注册表 + 接口 + 一条样例数据贯通整条管线),骨架可独立跑通、可测,内容后填。

## 8. 与阶段1 的衔接 & 搁置项
- 阶段1 的「静态造物台」= 本文 **L2 的最小子集**(房主从素材池组方块定义),不碰生存经济。
- 自定义 PNG 传输方案:**搁置**(素材走代码层)。重启条件 = 游戏内上传 / AI 管线。
- 生存制造(工作台/图纸/材料/库存)= **重型模式**范畴;轻量模式只读定义、可放置静态物。

## 9. 开放问题 / 待测
- 卡死真实上限:待实测(增量 / 定义槽 / 图集),测后写《房主须知》。
- 删除自定义方块时,已放置实例的回退策略(空气 vs 占位块)。
- 图纸在任务 / 探险中的获取分布与节奏。
- 管理员授予的 UI 与持久化(轻量 = 本地,重型 = 服务器账号)。
- 行为包库的初始规模——决定显现行为多样性的上限,**成本在开发者**。
