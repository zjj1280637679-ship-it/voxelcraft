## 0. 摘要与总法则

### 0.1 一句话

**内容只写能指(无限),逻辑只写解析器(有限),没人能伪造所指。**

整个内容系统是一张【实体—关系图】,图里存的全是**能指**(轻量、可哈希的引用/令牌);**所指**(真内容/含义/结果)由内核解析器**按需解析**。加内容 = 往图里加能指(零波及解析器);加解析能力 = 扩解析器(罕见,需评审)。

> **与之并列的第二总法则见 §0.6:机制中立(消极自由)**——基础设施只提供「能力 + 优雅降级」,绝不替策划做「应该 / 不得」的价值判断。第一法则管「内容怎么存、所指怎么防伪」,第二法则管「基础设施不许越权干涉什么」。

### 0.2 能指 / 所指(SIGNIFIER / SIGNIFIED)

| 概念 | 是什么 | 谁写 | 性质 |
|---|---|---|---|
| **能指 Signifier** | 轻量引用/令牌:对象 id、资产哈希、属性令牌、边数据 | 内容侧(god 在上帝模式编辑) | 可哈希、可校验、零真内容、可无限加 |
| **所指 Signified** | 真内容/含义/结果:贴图字节、压制百分比、掉落事件 | 没人直接写——由解析器按需算出 | 不可伪造(解析器纯函数 + 锁定字段 + 双端复算) |

**为什么「没人能伪造所指」**:所指从不被存储或传输,它每次都由内核解析器从能指现算。房主即便篡改能指,要么改动落在锁定字段上(物理无可写入口)、要么改动改了注册表哈希(连不进房)、要么成员用同一解析器复算后丢弃——伪造在数据层不可达(轻量模式诚实边界见 §3.4 与 §5.3 / §7,A 档可强制、C 档需重型服务器根治)。

### 0.3 一张图说清「寻址即一切」

```
┌─────────── 能指层 SIGNIFIER(内容侧,无限可加) ───────────┐
│  对象 id ──┐                                               │
│            ├─▶ 一条【定义 Definition】=                    │
│  令牌/引用 ┘     { 对象, 表现引用, 本质令牌, 关系边[] }     │
│                  (god 在上帝模式 L2/L3 编辑;只写、可哈希)  │
└────────────────────────────────────────────────────────────┘
                       │  寻址 / 解析(按需,内核做)
                       ▼
┌─────────── 解析器层 RESOLVER(逻辑侧,小而固定,需评审才扩)─┐
│  ① 资产解析器  assetId ─▶ 真实字节(PNG/模型/动作包数据)   │
│  ② 交互解析器  令牌+标签 ─▶ 净效果 effect(伤害/免疫/治疗) │
│  ③ 关系执行器  边     ─▶ 实际生成事件(破坏→掉落…)         │
└────────────────────────────────────────────────────────────┘
                       │  解析出的【所指】= 真内容/含义/结果
                       ▼
┌─────────── 实例层 INSTANCE(运行时,只引用定义) ───────────┐
│  运行时实例 = { defId(寻址回定义), 位置, 当前HP, 背包… }   │
│  迁移快照搬的是【实例】;定义靠 双哈希 双端同源不随快照搬   │
└────────────────────────────────────────────────────────────┘
```

### 0.4 两条贯穿全文的不变量

- **不变量①(锁定字段无可写入口)**:本质的【锁定本质】(tier / dropPool / harvest / combat / toughness)只住在只读原型注册表,显现 schema **根本没有能触碰它们的键**——这是「木鸡不能掉钻石」最物理的防线(阶段1 §2.2)。
- **不变量②(引用必过白名单 + 复验)**:显现引用的每个 assetId / packId / traitId 必须 `∈ allow.*`,且双端跑同一 `validateManifest` 复验(阶段1 §2.5 / §2.6)。

### 0.5 四个术语 + 两个分离

四术语,每个分能指/所指:**对象 Object / 表现 Presentation / 本质 Essence / 关系 Relation**(§1)。
两个分离贯穿落地:**定义 vs 实例**(§4)、**能指 vs 所指**(§0.2)。
规则(无免疫 / 阶级压制 / 采集速度 / 挖掘时间)是读本质的**全局内核函数**——交互侧现已统一为 `EffectResolver` + 标签修正数据(§3.2),无免疫是数据约定而非代码律;不是关系、不挂对象(§3)。

### 0.6 第二总法则:机制中立(消极自由)

**基础设施只提供「能力 + 优雅降级」,绝不替策划做「应该 / 不得」之类的价值判断**——那是策划的想象力自由,机制不得越权干涉。这条法则与 §0.1 并列,是本文裁掉一切「强加规则」的依据(房主拍板,2026-06-13)。

| 面向 | 边界 | 例 |
|---|---|---|
| **对策划开放**(god 上帝模式 L2/L3) | 机制不裁「合不合理」,只保证「不崩」。卡死归房主(造物 §3.3) | 可造**空产出**的边(赌博式合成,产出可一无所获);可造**循环 / 自动变换**;可**删**任意关系 / 动作包 / 对象 |
| **对玩家锁死**(原型锁定不变) | 玩家拿不到 god 权益、改不了锁定本质、伪造不了所指 | 不变量①②(§0.4)一格不松;消极自由**只**对创作层开放,游玩层照锁 |

- **越权的反面教材(本文据此「不写 / 撤销」)**:任何形如「`craft` 必须产出 ≥1」「某物**只能**世界生成」「**禁止**环」的强加规则,都是基础设施用自己的口吻替策划**否决了一类设计**——属机制越权。本文一律改用**值中立的边界**替代:空查 = 无事发生(不报错)、删除 = 优雅降级(不强制瞬死,§2.7)、防永动 = 守恒 + 体量上限(不禁环,§7 E1)。
  > 注:`craft≥1`、`world-only` 这两条曾在讨论中被提出,**从未写进任何文档**;此处明确登记为「不采纳」,防下游代理误补。
- **一句话判据**:机制只回答「**能不能做到 / 做不到时怎么优雅退化**」,**绝不**回答「**该不该这么设计**」。后者永远是策划的自由。

### 0.7 第三总法则:统一对象底座(分面可插拔)

承房主裁决(2026-06-13):**代码层只有一个对象依赖,全体对象(玩家/怪/Boss/方块/物品/工具/区域/捏人草稿)共用它;差异不是「不同类型走不同代码」,而是「拥有哪些分面」。** 这是 §0.1(能指/所指)与 §0.6(消极自由)推到对象层的必然结论。

**对象 = 不可分的身份核 + 一组「在则有、不在则无」的可选分面束**(facet bundle):

| 分面 facet | 内容(= 既有术语) | 缺席 = | 缺席对象例 |
|---|---|---|---|
| `present` 表现 | §1.2 Presentation(faces/skin/model/icon) | 不可见 | 纯逻辑触发体 |
| `essence` 本质 | §1.3.2 EssenceTune(tune/traits;锁定半边仍只在 PROTOTYPES) | 不参与规则结算 | 纯装饰物 |
| `relation` 关系 | §1.4/§2 RelEdge[] | 完全惰性(消失=纯消失,§2.7) | 不可交互背景物 |
| `behavior` 行为 | 动作包 pack | 无 AI = 呆立/静止(§2.7) | 方块、呆立生物 |
| `dynamic` 动态 | 声明「运行期需要哪些动态槽」(pos/hp/fsm/rng,不存值) | 无逐 tick 演进 = 可烘焙 | 静态方块/物品 |

**「没动作的方块 = 缺 behavior/dynamic 分面的对象」**——正是用户原话「方块和生物底层依赖一样,只是没动作」的数据化:动作 = **有** BehaviorFacet,而非 `behavior=null`。靠「分面键在不在」区分,根除 null 哨兵歧义。

**异体方块范式(present 分面变体的教科书例,房主裁决 2026-06-14)**:苔藓石头 = 石头**同一 protoId**(本质=石,tier=1)+ 仅 `present` 分面(外观)与名称不同。**通则**:凡「看着不同、本质相同」的方块一律 = 同基础材料 protoId + 异 present 分面 —— 零 tier 歧义、零采集偏置分叉,破坏出基础材料、拆除得变体块本身(§2.5.1)。据此**天然复合物子系统取消**(四基元代数 / 木级以下材料档,覆盖 WP §3.1–3.2;经济 §1.3 / §9.2 已决)。

**kind 降级为派生标签**:不再是一等存储字段。`kindOf(def)` 由「拥有哪些分面 + proto.matClass」加载期算一次并冻结,**仅供非承重消费**(UI 分类、调度粗分流、日志)。承重派发一律按 `has(facet)`,**绝不 `switch(kind)` 选代码路径**(review 红线):渲染看 `has(present)`、结算看 `has(essence)`、交互看 `has(relation)`、AI 看 `has(behavior)`、是否活成实例看 `has(dynamic)`。块和生物走**同一** `resolve()` 骨架,差别只在「块没有 behavior/dynamic 分面所以那两段派发跳过」。

**挂到 §0.6(能力槽视角)**:每个分面就是一项「能力」。基础设施给全体对象**铺满**所有分面的解析器(都够得着 = 用户原话「都要有等价接口」);某对象「有没有某能力」= 策划填没填那个分面(赋予/休眠);「显不显现」= 游戏层 × 权限 × 模式 三元门决定(其中**权限维 = §8.1 造-B11 的 capId 能力集判定**)。机制只管「能不能赋予 / 缺了怎么优雅降级」,**绝不裁「该不该赋予」**(§0.6)。

**捏人 = 造人**:同一造人管线、同一 ObjectDef、同一 `validateManifest`,差别只是捏人产物 present 受限皮肤集 + 一个「不入世」显现门——不是另一套 schema、不是另一条代码路径。
- **堵死玩家后门(闭合 C 灰区)**:捏人**对玩家开放**,但其产物**不写 contentHash 锁定的世界 DefinitionSet**(§1.1)、只活成玩家**本地角色档 / 本地 Instance**(present 受限皮肤集)。故「捏人对玩家开放」**不违 §0.6 游玩层照锁**——玩家经捏人管线**拿不到 DefinitionSet 写权**,塞不进 relation/behavior 进世界。
- 若某世界确要让捏人产物**入世界定义集**,则回退受 **§8.1 造-B11 的 capId 创作层访问控制**钳制(玩家无 god cap ⇒ 不可写)。
- **两层别混**:「不入世」显现门是**运行期/数据层**的存在性开关(管「显不显现」);capId 是**创作层**写权限(管「god 能不能写」)。二者正交、不是同义词。

**待拍点①裁决 · 混合冻结**:**分面【类型集】硬冻**(进 REGISTRY_HASH;加一个分面类型 = 加一条解析派发分支 = 扩解析器,需评审)**+ 分面【内字段】软可选**(进 contentHash;加字段由既有解析器消费、受 allow/budget 钳窗,零评审 = 内容无限的来源)。比纯软更严(防注入解析分支)、比纯硬更活(内字段仍可无限长)。

**待拍点②裁决 · 存储措辞**:存储形态不由类别决定,而由 **`has(dynamic)`** 决定——无动态分面 → 无演进态可搬 → 解析后塌缩**烘焙**成 block id 进 edits(不活成 Instance);有动态分面 → 活成 **Instance** lazy 重解析。这是**统一接口、特化存储**:方块与生物在定义层同一 ObjectDef、同一 resolve 骨架,差别仅「有无 dynamic 分面」;存储分叉是这个分面的**推导结果**,**不是两个类型、不是 kind 分支、也不是「零改」**。会动的方块 = 追加 behavior+dynamic 分面 → 自动从烘焙侧滑到 Instance 侧;呆立生物 = 有 dynamic 无 behavior(活成实例占名额但不跑 AI)。

**两条随本法则一并立的一等契约**(全文定义见 §3.6 / §3.7):
- **序(确定性遍历序)**:分面派发与序列化必须按**固定枚举序** `present→essence→relation→behavior→dynamic`;一切喂进 缄/掷/承诺 的集合按稳定下标序,**严禁 Map/Set 迭代序**(否则双端逐位失配 desync)。
- **隔(故障隔离)**:每个分面解析器**必须定义缺席/坏值降级分支**(缺 present→纯色占位、非法 essence→拒实例、坏 relation→边静默失效);单分面失败隔离为「该分面降级」,**绝不崩整对象/整帧**。

**SoA 红线(不可松)**:分面是**定义层的纯数据可选字段**,加载期解析后**塌缩**进扁平烘焙表 / AoS Instance;运行期内核**绝不**持有「按分面分桶」的列存 / archetype(§5.4)。`has(facet)` 派发 ≠ facet-bucketed storage。N≤16 用 AoS 一字不改。

**定位**:本节是**统一对象视图**(类比 §8「映射为视图」),**不推翻平行文档**——造物 / 阶段1 仍可用 kind / 四术语,本节只在底座层把它们收敛为分面。

## 1. 四术语的精确数据形状(各含能指 / 所指)

> 全部用 JS 伪代码,只定形状不写实现。刻意复刻引擎成熟模式「**有序数组,下标即稳定 id,永不复用,弃用置 deprecated**」(对齐 `constants.js` 的 `BLOCK`/`BODIES`/`PALETTE` 与阶段1 §2.2)。

### 1.1 对象 Object —— 能指=对象 id + 类别;所指=对象定义

```js
// ============ 能指:对象 id(寻址代号) ============
// objId 不是对象本体,只是「定义集数组的下标」。永不复用,弃用只置 deprecated。
type ObjId   = int                       // 0..N,下标即稳定 id
// ⚠ kind 不再是一等存储字段(§0.7 统一对象底座):它是 kindOf(def) 的【派生标签】,
//   由「拥有哪些分面 + proto.matClass」加载期算出,仅供 UI/日志/调度粗分流;承重派发只看 has(facet)。
//   下列枚举域保留,仅作 kindOf 返回值与 Prototype.kind 用。
type ObjKind = 'block' | 'material' | 'creature' | 'tool' | 'region'
//             方块      材料         生物         工具      区域

// ============ 所指:对象定义 Definition = 身份核 + 可选分面束(§0.7 统一对象底座) ============
// god 在上帝模式编辑的【模板】(造物 §4.1「每条定义=原型引用+显现」)。
// 关键:四术语在此收敛为「分面束 facets」,但【锁定本质】不在这里 —— 它在只读 PrototypeRegistry,
//      protoId 单向引用(不变量①:无锁定字段可写入口)。kind 字段已移除 → 见末尾 kindOf(def)。
type DynField = 'pos' | 'hp' | 'fsm' | 'rng' | 'state'   // dynamic 分面声明的运行期动态槽(只声明、不存值;值在 Instance)
Definition = {
  objId,                  // 能指:寻址代号(= 数组下标)
  protoId,                // → 指向锁定原型(本质·锁定部分唯一来源,见 §1.3)
  facets: {               // ★分面束:键在=有该分面 / 键缺=无该能力。绝不写 null 占位(键缺即语义,根除哨兵歧义)
    present?:  Presentation,            // 表现(§1.2);缺 → 不可见(纯逻辑触发体)
    essence?:  EssenceTune,             // 本质·可调附性(§1.3.2,tune+traits);缺 → 不参与规则结算
    relation?: { edges: RelEdge[] },    // 关系(§1.4/§2);缺 → 完全惰性(消失=纯消失,§2.7)
    behavior?: { pack: PackId },        // 行为:动作包(∈ allow.actionPacks);缺 → 无 AI=呆立/静止
    dynamic?:  { fields: DynField[] },  // 动态:声明运行期需哪些动态槽;缺 → 无 tick 演进=可烘焙(§4 存储分叉判据)
    // 加分面【类型】= 扩解析器需评审(进 REGISTRY_HASH);加分面【内字段】= 零评审(进 contentHash)。§0.7 混合冻结
  },
  name,                   // 1..16 码点,过 cleanName,仅展示(显现自由)
  deprecated: false,      // 弃用墓碑(永不复用 id;造物 §6 删除语义)
  contentHash,            // 本定义对「能指内容」的哈希(对齐造物 §7.3)
}
// kindOf(def) = 由 facets 形状 + proto.matClass 派生的只读标签(§0.7);承重代码只读 has(facet),从不读 kindOf


// ============ 容器:世界定义集 DefinitionSet(造物 §4.1) ============
// = 内置定义 −(房主删除)+(房主/管理员新增)。可变层,由 contentHash 锁。
DefinitionSet = {
  defs:        Definition[],   // 有序,objId=下标
  builtinCount,                // 前 N 条为内置(随版本发布);N.. 为房主新增
  contentHash,                 // 整集哈希(造物 §7.3 (b);握手比对,§6.1)
  registryHash,                // = REGISTRY_HASH(造物 §7.3 (a)+阶段1 §2.7;锁代码层注册表)
}
```

> **为什么对象的所指是「定义记录」而非「真东西」**:真东西(贴图字节、压制结果、掉落事件)由三类解析器在运行期算出。Definition 只记下「去哪寻址」——它本身仍是纯能指(可哈希、可校验、零真内容)。

### 1.2 表现 Presentation —— 能指=资产引用;所指=真实资产字节

资产 id = **内容哈希**(降采样 32×32 RGBA 的 SHA-256 前 16 字节 = 32 字符 hex,对齐阶段1 §3.2 + 造物 §7.1),**绝不可改用自增**。动作包是**数据非代码**,内核只有固定行为原语 `prim`。

```js
// ============ 能指:资产引用(全是哈希/下标,不含字节) ============
```

### 1.4 关系 Relation —— 能指=边数据;所指=实际生成事件(关系执行器解析)

关系**不另存为独立对象**:它就是「寻址」本身(正查"我变成什么"、反查"谁产生我")。边数据是轻量令牌四元组,所指(真实生成事件)由关系执行器按需算出。详细的边数据形状、双向索引、门表达见 §2;此处给概念形状。

```js
// 能指:边数据 RelEdge(轻量四要素 {触发, 门, 输入, 输出}),挂在 facets.relation.edges(§1.1;旧扁平 Definition.relations 已并入 relation 分面)
RelEdge =
  // ① 变换边(动作触发:破坏/制造/击杀/放置 → 消耗输入、产出输出)
  | { kind:'transform', trigger:'break'|'craft'|'kill'|'place'|'react',
      gate:{ workbenchTier?, blueprint?, toolTier?, matNature? },  // 门:不消耗,只校验「够不够格」
      input:  { id:ObjId, consume:int },                          // 消失/消耗
      output: [{ id:ObjId, min, max, chance }] }                  // 生成(走 dropPool 风格)
  // ② 分布边(世界生成触发:区域条件 → 寻址相关实体 → 按密度生成;空查=不生成)
  | { kind:'distribute', trigger:'worldgen',
      gate:{ biome?, tierMax?, yRange? },
      output:{ id:ObjId, density } }
```

> **关系为什么「不另存对象」**:每条边只是 `{触发,门,输入,输出}` 四个能指字段。"破坏苔藓石头掉什么" = 正查该对象 `trigger:'break'` 的边;"谁能造钢" = 反查所有 `output.id==钢` 的 `craft` 边。关系执行器是无状态纯函数,所指算完即抛,只写进**实例**(背包/世界),不回写定义。

## 2. 关系层:变换 + 分布 + 寻址正反索引

> 本节把 §1.4 的概念形状落成精确边数据、双向索引、门表达、删除处置。**边内联在原型/定义里,随两道哈希闸门覆盖,不进线格式**;反向索引是派生数据(加载期建,不落盘、不进迁移帧),与 `world._editsByChunk`(world.js:32-34 派生索引)同模式。

### 2.1 统一边数据形状(判别式联合)

```js
// 边 = 能指集合,全字段是引用(int 下标 / 字符串哈希 / 令牌枚举),无内联内容。
// 内联在 PROTOTYPES[protoId].edges:[...](或 Definition.facets.relation.edges);下标即 localEdgeId(稳定,弃用置 dead)。
Edge = {
  // ---- 共用头 ----
  kind,            // 'xf' | 'dist'  —— 判别式
  trig,            // xf: 'break'|'dismantle'|'craft'|'kill'|'place'|'react';dist: 'worldgen'
                   //   dismantle = break 的无损变体(产出块本身而非材料);调度与降级见 §2.3 / §2.5.1
  gate,            // 门(R2.3 / §2.4);无门 = null。dist 的「门」走 cond
  // ---- 变换专属(kind:'xf')----
  inp,             // 输入:[{ ref, kind:'proto'|'block'|'tag', n, fate:'consume'|'vanish' }]
  out,             // 输出:[{ ref, kind:'proto'|'block'|'item', n, chance, by? }]
  cross,           // true=唯一合法越级(仅钢:craft 产物 tier>主体 tier,经济 §4.2 Q24),仅 matNature=crafted 允许
  by,              // RNG 来源:'det'(确定性必出) | 'roll'(掷点,可复验)
  // ---- 分布专属(kind:'dist')----
  cond,            // 区域条件:{ biome:[..], yBand:[lo,hi], tierMax?, near?, density, cluster? }
                   //   tierMax = 该区域允许生成的实体 tier 上限(对齐 §1.4 RelEdge.gate.tierMax;E8 防恶性误匹配核心字段)
  spawn,           // 生成:{ ref, kind:'proto'|'block', n:[min,max] }
  // ---- 共用尾 ----
  dead,            // true=弃用(R7),索引跳过,能指号永不复用
}
```

- **边的能指号** = `(protoId, localEdgeId)` 二元组,`localEdgeId` = 该原型 `edges` 数组下标(复刻「下标即稳定 id」铁律)。全局边句柄 `EH = (protoId<<10)|localEdgeId` 走 **32 位有符号位运算**(JS `<<` 在 32 位上做,到不了 2^53),故 `PROTO_COUNT` 须 `< 2^21`(约 200 万,protoId 占 bit10..30 即触及符号位 bit31 之前);加载期静态断言 `protoId < 2^21` 且每原型 `edges < 2^10`(≤1024 边)。**若未来突破 2^21,改用乘加式句柄 `EH = protoId*1024 + localEdgeId`(走 2^53 安全整数,不依赖 32 位位运算)。** 句柄只内部用,不进线格式。
- **`inp`/`out` 用 `ref+kind` 而非裸 id**:支持「具体原型」(`proto`)、「方块 id」(`block`,过 isKnownBlockId)、「标签集」(`tag`,如 `tag:'wood_tier'` 寻址全体木级天然材料)。`tag` 让一条边覆盖一类对象,是「内容只写能指(无限)」的关键收敛点。
- **多元组预留**:`out`/`inp` 一律设计成「集合 + 数量 + 概率」位(对齐 Minecraft loot table 的 weight/set_count),后续表达「1 矿石+燃料→1 锭+概率渣」零波及解析器(留更改后路)。

### 2.2 正反双向寻址索引(加载期一次派生,运行期零计算)

```js
// forward:对象 → 它的全部出边(我作为动作主体/输入时会触发什么)
FWD = Map< protoId, { break:[EH...], dismantle:[EH...], craft:[EH...], kill:[EH...],
                      place:[EH...], react:[EH...], worldgen:[EH...] } >
//   每 protoId 的记录须预铺全部 trig 槽(含 dismantle),否则 buildEdgeIndex 的
//   FWD[protoId][e.trig].push 会写到 undefined;dispatchMine 读 'dismantle' 槽也才得空数组而非 undefined
// backward:对象 → 它的全部入边(谁产生我 / 我是谁的输出)
BWD = Map< refKey, [EH...] >          // refKey = `proto:<id>` | `block:<id>` | `item:<id>`
// 分布二级聚类(避免每区块全表扫描)
DIST_BY_BIOME = Map< biomeTag, [EH...] >
```

**构建算法(纯函数,确定性)**:

```
buildEdgeIndex(PROTOTYPES, defs):
  FWD={}; BWD={}; DIST_BY_BIOME={}
  for protoId, proto in PROTOTYPES:
    for localEdgeId, e in proto.edges:
      if e.dead: continue                          # 弃用边不进索引(R7)
      EH = (protoId<<10)|localEdgeId
      if e.kind=='xf':
        FWD[protoId][e.trig].push(EH)
        for o in e.out: BWD[refKey(o)].push(EH)
        # 阶级上限断言【按 trig 分流】(发布期承重防线,阶段1 §2.2 的单一权威口径):
        #   · break/dismantle/kill/react(主体即被采集/拆除/击杀/反应物)→ 上限锚【主体原型 tier】:
        #       ∀ o in e.out: tierOf(o.ref) ≤ proto.tier
        #       (dismantle 产出=块本身,tierOf(out)=tierOf(主体),断言天然成立)
        #   · craft(主体=工作台,门=台/图纸不消耗、不计入)→ 上限锚【被消耗材料 tier】:
        #       ∀ o in e.out: tierOf(o.ref) ≤ max( tierOf(i.ref) for i in e.inp if i.fate=='consume' )
        #       —— 既封死「低料造高料」零成本回路,又放行「低台 + 高料 → 高成品」(经济 §4.2 Q24、造物 §5)。
        #   唯一例外:e.cross==true(仅钢)→ 跳过上述断言。cross 仅表「成品 tier > 输入材料 tier」这一更窄的越级
        #            (制造线封顶处),不表「成品 tier > 台 tier」(后者本就由 craft 锚材料 tier 而合法放行)。
      else: # dist
        FWD[protoId].worldgen.push(EH)
        for b in e.cond.biome: DIST_BY_BIOME[b].push(EH)
        BWD[refKey(e.spawn)].push(EH)
        # dist(主体即生成物来源)→ 上限锚【主体原型 tier】:tierOf(e.spawn.ref) ≤ proto.tier
  freeze(FWD, BWD, DIST_BY_BIOME)
```

**查询接口(关系执行器唯一入口)**:

```js
fwdEdges(protoId, trig) -> [Edge...]   // 正查:我发生某动作 → 候选出边,O(1)
bwdEdges(refKey)        -> [Edge...]   // 反查:谁能产出此 ref(配方溯源/掉落溯源/平衡审查),O(1)
```

**空查语义统一(铁律)**:查不到键 → 返回**空数组** → 「不生成任何事件」。变换空查、分布空查(R5.3)、采集无匹配边 同义——**寻址失败永远是「无事发生」,绝不报错、绝不兜底生成默认物**(防显现伪造「凭空产出」)。"谁掉钻石" = 反查 `BWD['item:diamond']`(平衡审查关键查询,确认木鸡不在其中)。

### 2.3 触发器令牌表(trig,有限枚举,扩它=扩解析器需评审)

| trig | 语义 | 触发时机(落地锚) | 输入 fate |
|---|---|---|---|
| `break` | 破坏方块/采集(有损,产出**材料**) | 玩家挖掘命中(world.raycast,world.js:180) | 输入方块 vanish |
| `dismantle` | 拆除方块(无损,产出**块本身**) | 玩家持**拆除工具**挖掘命中;无此工具 → 调度降级为 `break`(§2.5.1,= Silk Touch 思路) | 输入方块 vanish |
| `craft` | 工作台制造 | 三门齐备 AND(造物 §5) | 材料 consume |
| `kill` | 击杀生物 | 内核 `hp<=0`(阶段1 §1.6) | 实体 vanish |
| `place` | 放置方块/物 | 玩家放置(host.js:105 block 入站) | 手持物 consume |
| `react` | 环境反应(条件触发,tick 检测) | 世界 tick 检测(水+岩浆→黑曜石,经济 §1.3) | 触发物按边定义 |

> **拆除→破坏的优雅降级(消极自由 §0.6)**:`dismantle` 不是「另一种挖掘」,而是 `break` 的**无损变体**——持拆除工具走 dismantle 边(产出块本身),否则**降级**走 break 边(产出材料)。机制不禁止「无工具拆除」,只优雅退化,与 Minecraft Silk Touch 同思路。派生调度见 §2.5.1。
>
> **自动/周期变换:不禁止,以守恒 + 体量上限兜底(取代禁环,见 §0.6 / §7 E1)**。`react`(条件触发、世界 tick 检测)已是 tick 驱动;水流等流体类自动变换同属此类。**流体规则当前取 MC 风格占位**(无限源 + 蔓延),封装在可换的 **`FluidRule` 接口**背后;**守恒版(消失+移动)延后**——日后「改逻辑」= 写一个新规则原语实现(评审一次)+ 翻数据选择子,而**引擎(tick 循环 / 活跃前沿集 / 双缓冲定序§3.6 / 体量上限)规则无关、只建一次**(「换规则 = 接口 + 数据选择子;新原语罕见需评审」,§0.1/§3.5;液体是「会 tick 的方块」=第三种存储归宿:在世界网格 edits、却每 tick 更新,引擎机械待 §5 骨架)。基础设施**不替策划禁掉循环或自动转化**(那是机制越权),而以三道**值中立**边界兜底:① **阶级单调**封死零成本升级回路;② **守恒**——自动/流体变换质量入=出(「下面加多少水上面减多少水」,刷不出净增物);③ **体量上限**——活体 ≤16 硬钳 + 每区域/帧方块生成预算可配(「控制水体生成的多少」)。三者合起来让「无限增生」物理不可达,**无需基础设施对设计拓扑下「禁止环」的 judgment**。
>
> 门 gate(工具/台/图纸)**绝不出现在 `inp`**(门=不消耗,造物 §5)。

### 2.4 门 gate 的表达(数据,非函数)

门是**可组合谓词的数据化**(造物 §7.2 把 Gate 设计为函数 `(ctx)=>bool`;本文把它**降为数据**由有限内核谓词解析——这是对造物的收窄优化,见 §8 造物 B-6)。

> **2026-06-14**:**采集/破坏的「效率」(原 `toolTier` 软门)已移入 EffectResolver(§3.2)**——效率是标签修正数据、不再是门。§2.4 现在只管**硬门(权限类)**:`benchTier`/`blueprint`(craft 准入)、`toolKind`(dismantle 调度)。下面 `toolTier` 行作废,保留仅为对照。

```js
gate = [                                          // AND 语义;空数组/null = 无门,永真
  { g:'toolTier',  tier:int },                    // 工具阶级门:采集/破坏(软门)
  { g:'toolKind',  kind:string },                 // 工具种类门:拆除需 'dismantler'(硬门;不满足→调度降级 break,§2.5.1)
  { g:'benchTier', tier:int },                    // 工作台阶级门:制造(硬门,天然料)
  { g:'blueprint', bp:int },                      // 图纸门:制造(硬门)
  { g:'matNature', allow:'natural'|'crafted' },   // 天然/人造判别(配合越级规则)
]
```

**软门 / 硬门二分(落地最易写错处,必须在 gateOk 解析器显式分流)**:

| 门 | 解析读什么 | 判定(出处) | 软/硬 |
|---|---|---|---|
| ~~`toolTier`~~(作废) | — | **已移入 EffectResolver(§3.2)**:采集效率 = `target.tier` 标签修正数据,正负叠加、可专精免疫/治疗 | 不再是门 |

### 2.5 变换边执行流程(原子单事务)

```
applyTransform(actorProto, trig, ctx):            # ctx={tool,bench,blueprints,pos,rng,...}
  edges = fwdEdges(actorProto, trig)
  if edges.empty: return []                         # 空查=无事发生(R2.2)
  events = []
  for e in edges:
    if not gateOk(e.gate, ctx): continue            # 硬门不过→跳过(软门只调系数不否决)
    # —— 预检(可落性):输出 ref 已知 isKnownBlockId/未弃用、目标格空、实体名额够 ——
    if not preflight(e.out, ctx): continue          # 任一输出不可落 → 整条边 no-op(防凭空蒸发,§7 E7)
    # 提交原子性:预检全过(上面 preflight)才一次性 consume+produce,否则整条 no-op。
    # 注意:原子性 ≠「多概率输出全有或全无」—— 各 by:'roll' 项各自独立掷点,掷不中只是【该项不产出】,不回滚已消耗输入。
    consumeOrVanish(e.inp, ctx)                      # 按 fate 一次性消耗/消失输入(提交点)
    for o in e.out:                                  # 多概率输出:逐项独立掷点(非全有或全无)
      hit = (e.by=='det') ? true : rollChance(o.chance, ctx.rng)   # mulberry32 同一子种子,逐项独立、可复验
      if hit: events.push({ produce:o.ref, kind:o.kind, n:rollCount(o.n,ctx.rng), src:EH })
  return events                                      # 房主 cast 给成员;成员同 rng 复算复验
```

> **空 `out` 合法(消极自由 §0.6)**:`out:[]` 是合法边(赌博式合成:消耗输入、概率性一无所获)。`preflight([])` 真空通过 → `consume` 照常 → 不产出任何事件即返回。这**绝非错误**,机制不替策划强加「必须产出 ≥1」。空 `out` 与「掷点全不中」殊途同归——都只是「这次没产出」,不回滚已消耗输入(原子性约束的是「消耗↔产出同生共死」,不是「保证有产出」)。

**与既有 cast 模型对齐**:`block` 类产出 → `world.setBlock` → 进 edits Map → `net.cast({t:'block'...})`(host.js:114-119),**零新协议**;`item/ent` 类产出走阶段1 §2.3.2 `ent.*` 族或背包消息。关系执行器只产「已结算事实」,由现有传输层投递。

### 2.5.1 破坏 / 拆除调度(优雅降级,消极自由 §0.6)

玩家「挖掘」是**一个**动作,落到 `break` 还是 `dismantle` 由调度器按工具**优雅选择**——不报错、不卡操作、不禁止:

```
dispatchMine(actorBlockProto, ctx):                 # ctx.tool.kind / .tier 来自手持工具
  if ctx.tool.kind=='dismantler'
     and fwdEdges(actorBlockProto,'dismantle').some(e => gateOk(e.gate, ctx)):
    return applyTransform(actorBlockProto, 'dismantle', ctx)   # 无损:产出块本身
  return applyTransform(actorBlockProto, 'break', ctx)         # 降级:产出材料(有损)
```

- **破坏 break(有损)**:默认形态,方块 vanish,产出**基础材料**(苔藓石头 → 石头,丢苔藓外观;异体方块破坏 = 出基础材料,§0.7 / 经济 §1.3)。
- **拆除 dismantle(无损)**:持拆除工具时,方块 vanish,产出**块本身**(苔藓石头 → 苔藓石头块),= Minecraft Silk Touch。
- **降级而非禁止(§0.6)**:无拆除工具时**自动落到 break**,不报错。机制只提供「有工具 → 无损」的能力 + 「无工具 → 有损」的优雅降级,绝不下达「没工具就不准挖」的 judgment。
- **拆除是策划逐条赋予、非全局默认**:若某方块**没有** dismantle 边(策划没给),持拆除工具也只能 break(空查 = 无事发生,R2.2)。哪些方块可无损取回,是策划的设计自由。
- **拆除不改经济结算**:采集门槛(工具−目标)/ 阶级压制 / 韧性照旧只看工具与方块阶级(经济 §2.2);dismantle **只换产出形态(材料 ↔ 块)**,不新增阶级路线。

### 2.6 分布边执行(区域条件 + 密度 + 空查)

```
distributeChunk(cx, cz, worldSeed):
  biome = biomeOf(cx, cz, worldSeed)                # 确定性地形
  cands = DIST_BY_BIOME[biome] ?? []                # 区域过滤(空 → 不生成,R5.3)
  results = []
  for e in cands:
    if not condMatch(e.cond, cx, cz, biome): continue
    # 区域 tier 上限钳(E8 恶性误匹配防御):cond.tierMax 给定且生成物超限 → 按密度 0 跳过
    if e.cond.tierMax != null and tierOf(e.spawn.ref) > e.cond.tierMax: continue
    rng = mulberry32(hash2(cx, cz, worldSeed, EH))  # 区块×边 确定性子种子
    count = sampleDensity(e.cond.density, e.cond.cluster, rng)
    for i in count:
      pos = samplePos(cx, cz, e.cond.yBand, rng)     # surfaceHeight 仅在已加载半径内调(阶段1 §1.3)
      results.push(e.spawn.kind=='block' ? blockSpawn(pos,e.spawn) : entSpawn(pos,e.spawn,rng))
  return results
```

**误匹配防御 = biome × tier 闸门**:分布 cond 引用原型的 `biome`(白名单)+ 新增 **`cond.tierMax`** 字段(区域 tier 上限,对齐 §1.4 RelEdge.gate.tierMax)。`condMatch`/`distributeChunk` 显式按 `cond.tierMax` 钳生成:`tierOf(spawn) > cond.tierMax` → 按密度 0(新手区高 tier 实体即便 biome 匹配也不生成)——把「钻石怪刷新手区」从「能配出来」变成「配出来也生成 0」(§7 E8)。**注意**:加载期 `tierOf(spawn) ≤ proto.tier` 断言(§2.2)约束的是「产物 tier ≤ 生成它的主体原型 tier」,**不能表达「此区域只允许 ≤ 某 tier 实体」**——后者必须靠 `cond.tierMax` 这一新增字段,二者职责不同、缺一不可。密度上限受 ≤16 活体硬钳;分布在 `ensureChunk` 一次性跑、不循环 tick。

**溯源链(验证寻址反查)**:黑曜石「水+岩浆」是 `react` 变换边(世界 tick 触发),「岩浆分布」是 dist 边;反查 `BWD['block:obsidian']` → react 边 → inp 含 lava → 反查 lava 的 dist 边 → volcano biome。

### 2.7 删除 = 按粒度优雅降级(消极自由 §0.6;造物 §6 删除语义 + §9 落点)

**删除不是「一律强制瞬死」,而是按删除粒度优雅降级**(房主拍板,2026-06-13)。关键洞见:**「删一条关系」≠「删寻址本身」**——边删了,protoId / 令牌还在;「对象消失会留下材料」本身就是**一条关系边**,删了它不等于删了对象的寻址。据此分级(分面视角见 §0.7:四档 = 分面逐级移除——删 relation 分面→惰性、删 behavior 分面→呆立、删 dynamic→静态化、删整 protoId→瞬死):

| 删什么(粒度) | 后果 | 为什么(消极自由) |
|---|---|---|
| 删**一条关系边**(如「消失 → 留材料」) | 该交互消失,**实例照常存在** | 边只描述世界事件,不承载实例存在性;删边 = 少一种事件,不是删对象 |
| 删**对象的全部关系** | 对象仍在但完全惰性:**「消失 = 纯消失」**(不留任何痕迹) | 「留材料」是关系,全删即触发「消失」却无产出——是策划的合法设计,不是错误 |
| 删**动作包**(`m.a` 指向) | 生物**呆立**(无 AI、不动,**仍渲染 / 仍可交互**) | proto 仍可解析锁定本质,只是行为数据没了 —— **呆立 ≠ 死** |
| 删**整个 protoId / 方块** | 实例孤儿读不到锁定本质 → 触发下表悬空处置 | tier/combat 读不到,实例无法继续存在,才走瞬死 / 占位 |

**只有最后一档(删整个对象)才触发下面的悬空处置;前三档是组件级优雅降级,实例不死。** 房主 L3 删某原型/方块后,统一处置接口 `reapDanglingEdges(deletedRef)`:

| 悬空类型 | 检测(靠 BWD/FWD) | 处置 |
|---|---|---|
| ① 被删对象的**出边** | `FWD[deletedProtoId]` 全部边 | 整体置 `dead:true`,索引跳过 |
| ② 指向被删对象的**入边** | `BWD[refKey(deletedRef)]` | 逐边裁剪 `out`/`spawn` 的死 ref 项;裁空则整条置 dead |
| ③ 已**刷实例**引用被删**整个 protoId** | 运行时实例扫描 | 生物:无锁定本质可解析 → **瞬死**(造物 §6);方块:**渲染占位 + 交互降级为空气**(§7 E2 裁决,回填造物 §9)。**注**:删**动作包**(非删 proto)只致**呆立**不致死,见本节上表第三档 |

```js
reapDanglingEdges(registry, deletedRef) -> {
  registry',          // 出边 dead、入边裁剪后的新注册表(随之改哈希)
  killInstances:[entId...],   // ②③ 瞬死实例(房主 cast ent.die)
  retileBlocks:[blockId...],  // ③ 回退占位的自定义方块
}
```

> **删除即改哈希**:边内联在注册表 → 删/裁边 → 哈希变。运行期房主 L3 删的是 **DefinitionSet 层**(由 `contentHash` 锁),走同一接口但只重铸 `contentHash`、广播瞬死/回退事件(§5.7 / §7 E4)。代码层注册表的增删属构建期/版本期(改 `REGISTRY_HASH`)。

## 3. 解析器(三类:资产 / 交互 / 关系,小而固定)

三类解析器把能指翻译成所指,是「逻辑只写解析器(有限)」的全部。**加内容零波及它们;扩它们(加令牌/触发器/原语)罕见且需评审。** 全部纯函数,禁 DOM/定时器/`Math.random`/`Date.now`(阶段1 §1.2)。

### 3.1 解析器① 资产解析器 AssetResolver(能指 assetId → 所指 字节)

```js
interface AssetResolver {
  has(assetId): boolean,                 // 本地是否已持有字节
  get(assetId): Bytes | null,            // 取字节(未持有→null,触发按需分片拉取)
  resolveTile(tileRef): UV,              // 瓦片引用 → 图集 UV(贴图=关时回退纯色 tile)
  resolvePack(packId): ActionPack,       // 动作包 id → 只读状态机数据(纯数据,非代码!)
  version(): string,                     // 库版本(握手比对)
}
// 留后路:「游戏内上传/AI 管线」= 在 assetId 接口背后多挂一个来源,消费方零改动
//        (前提:产出字节必须按同一内容哈希铸 id,造物 §7.1)。
// 兜底:get 查不到 → resolveTile 返回确定性纯色占位(色相由 assetId 前若干字节哈希到 PALETTE),
//        全房同色不抖动;降级显示但保留实例(§7 E3 非对称兜底)。
```

### 3.2 解析器② 交互解析器 EffectResolver(统一:伤害 / 挖矿 / 治疗 / 补块)

> **2026-06-14 重大收敛(房主拍板)**:原"规则词汇"的 `suppression`(压制)+ `harvestGate`(采集门槛)+ `harvestBias`(偏置)**三个内核函数塌成一个 EffectResolver + 数据**。**不再有 tier 代码公式、不再有压制/采集专用函数**——等级、工具契合、克制、抗性、治疗、补块、针对某玩家,**全是同一个池子里的 id 标签修正(数据)**。这是「逻辑只写小解析器、内容全是数据」在交互侧的彻底兑现。

**核心:一次交互 = 一次「基础 × (1 + Σ命中的标签修正)」**

```js
// 目标带 id 层级标签集;攻击者/手持/附魔/(目标侧)护甲 各带「标签→修正%」数据包(全是数据,非代码)
effect(actor, target, ctx):
  base = power(actor)                         // 原始强度(=槽位×等级,WP §8.1;无工具=身体强度)
  mods = collectMods(actor, target, ctx)      // 攻击者+手持+附魔+目标护甲 的全部 {标签→修正} 条目
  sum  = Σ{ m.val : m.tag ⊑ target.tags }     // ⊑ = 前缀匹配(id 标签,见 §3.2.1);命中的修正【相加】
  aff  = clamp(1 + sum, FLOOR, CAP)           // FLOOR/CAP 是数据(默认 FLOOR 允许负=治疗,见下「无免疫」)
  return base * aff                           // aff>0 伤害 / =0 免疫 / <0 治疗(负伤=加 HP)
// 逐击施加到 target.HP(方块=toughness、生物=maxHp);HP≤0 → 走该目标的死亡输出边(块=break/dismantle 掉料、怪=kill 掉落)
```

- **tier 是数据,不是公式**:木剑的包里逐条写 `{石级:−50, 铁级:−75, 金级:−90, 钻石级:−95}`。**没有 `suppression(defTier,atkTier)` 函数了**——白皮书那张 45/15/5/1 曲线**降级为「默认 tier-亲和数据包」**(随版本内置,per-武器可覆盖,经济 §2.1)。
- **克制 / 抗性 / 针对玩家 同理,全是数据包**:屠龙剑 `{生物:−100, 生物龙:+200}`(对普通生物零伤、对龙双倍);护甲 `{火:−50}`(目标侧减伤);「针对玩家 zz」`{玩家.<zz的uid>:+50}`。**零新代码**。
- **加法 + 前缀匹配**:命中的修正**相加**(−50%生物 + +100%生物龙 = 净 +50%);深标签层层叠(金龙命中 生物/龙/金 三层)。加法天然 **order-independent**(不碰序契约 §3.6)。
- **affinity 的符号 = 交互种类(自动涌现)**:`>0` 伤害、`=0` 专精免疫、`<0` 治疗(负伤=加 HP)。**治疗 = 负 affinity 的攻击,无独立治疗系统**;给方块负 affinity = **修复受损方块**(补块)。木铲打钻石龙叠过 −100% → 负 → **奶它(被按摩)**。
- **无免疫 = 数据纪律,不是代码 floor**:对口工具靠「默认包别写到 −100% 以下」保持正伤(木剑对钻石停在 −95% = 还有 5%,可慢磨);错口工具 / 专精武器可叠到负 = 免疫/治疗。**FLOOR 本身是数据**(想让某 tier 也硬保底,就把 FLOOR 设成 ≥1%)——完全 §0.6 消极自由:机制提供可钳的下限,填多少是策划自由。这**超越**早前「无免疫=差距只减速不归零」的代码-floor 裁决(经济 §2.5 已同步)。
- **HP / 韧性仍是锁定本质**:`toughness`(方块 HP / 挖掘耐久)、`maxHp`(生物 HP)是锁定令牌(§1.3.1),被本解析器消耗;**挖掘耗时 = toughness × 形态系数(物理)**,与 affinity(效率)物理隔离——一个算「打掉多少血」、一个算「每击多久」。
- **制造门是另一回事**:工作台/图纸的 craft 准入(benchTier/blueprint,§2.4 硬门)是**权限能力**、不是 effectiveness——不进本解析器,仍是 gate 谓词。

```js
interface EffectResolver {
  effect(actor, target, ctx): float,             // 单次交互净效果(正伤/零免/负疗),逐击施加
  toughness(target, morphology, state=1.0): float,// 目标 HP/挖掘耐久(锁定 base × 形态 × 状态);物理隔离于 effect
  collectMods(actor, target, ctx): {tag,val}[],  // 汇集攻击者+附魔+护甲的 {标签→修正} 数据条目(纯数据)
}
```

### 3.2.1 标签命名空间(id 寻址·层级·锁版本;敲定开放问题 G5)

EffectResolver 的「前缀匹配」建立在一套**标签命名空间**上。**铁律:标签靠 id 寻址,绝不用名字串**(= §0.1 总法则 / §2.2「名字可重·寻址靠 id」/ §7 E10 在此重申)——否则两个同名对象、或名字撞标签,就出「同义字乌龙」。

- **标签 = 层级 id 路径**:`生物.龙.金` / `方块.金属.金` / `钻石级` / `玩家.<uid>`。每段是稳定 id(下标/uid),不是显示名。
- **前缀匹配 `⊑`**:修正键 `生物` 命中所有 `生物.*`;`生物龙` 只命中龙。对象带最具体的标签,前缀自动给它所有上层修正。
- **玩家名自动并入**:玩家「zz」内部 = 标签 `玩家.<uid>`;「zz」只是 present 显示数据;「针对 zz 的魔法」瞄的是 uid。两个 zz = 两个 uid = 不乌龙。(社交注:针对玩家是 grief 向量,服务器开关,§0.6。)
- **版本锁**:内置标签树进 REGISTRY_HASH(代码层);god 运行期加的标签进 contentHash(世界层),与 §0.7 分面类型集同闸门。名字永不进哈希、永不参与匹配。
- **确定性**:前缀匹配 + 求和按标签 id 稳定序(§3.6 序),双端逐位一致;加法本身 order-independent。
- **与 RelEdge.tag 共用同一标签树**:§2.1 关系边的 `tag:` 寻址(如 `tag:'wood_tier'`)与本命名空间是**同一棵树**(`wood_tier` = 路径 `材料.木` 的别名)——边引用用精确匹配、EffectResolver 用前缀匹配,语义一致(§6「tag 合并须确定性序」同源)。不另起一套寻址。

### 3.3 解析器③ 关系执行器 RelationExecutor(能指 边 → 所指 生成事件)

```js
interface RelationExecutor {

  backward(objId): RelEdge[],                  // 反查:扫全集 output.id==objId 的边(寻址即关系,不另存索引)
```

### 3.4 「没人能伪造所指」由三解析器如何保证

- **资产解析器**:assetId=内容哈希,房主下发字节必须重算 SHA-256==id,否则丢弃(自带防篡改)。
- **交互解析器**:纯函数;标签修正包走 contentHash 锁(房主篡改→哈希变→对账失败丢弃),锁定 HP/tier 令牌只读、双端同源。
- **关系执行器**:掉落走 seeded RNG,成员用同 `sd` 复算,对不上即丢弃。

三者合起来:**轻量模式做到「房主无法让诚实成员相信伪造的所指」**(A 档);阻止作弊房主本地自肥需重型服务器(C 档,阶段1 §2.6 诚实边界)。

### 3.6 一等契约:序(确定性遍历序)

随 §0.7 立。**一切喂进 缄(哈希)/ 掷(RNG)/ 承诺(contentHash·REGISTRY_HASH) 的集合,必须按稳定下标序遍历,严禁 Map/Set 迭代序。**

- 分面派发与序列化按**固定枚举序** `present→essence→relation→behavior→dynamic`(分面是有序键集、非无序 map;加载期断言遍历序固定)。
- 多输出按 dropPool 数组下标序(§5.5 已有);tag 合并按确定性序(§6);实体迭代不依赖 `Map<entId>` 迭代序(§5.5 已警告)。
- **判据**:任何「集合 → 哈希 / RNG / 线格式」的喂入若依赖插入序或哈希序,即 desync 隐患(review red flag)。这条横跨 缄/掷/封/闸门 四原子,故升一等契约。

### 3.7 一等契约:隔(故障隔离)

随 §0.7 立。**单个能指 / 分面解析失败,必须隔离为「该单元优雅降级」,绝不崩整对象 / 整帧。**

- 每个分面解析器**必须定义缺席 / 坏值分支**(加分面类型时评审清单的一项):缺 present→纯色占位(E3)、非法 essence→**拒整实例**(不变量①,decode 跑 validateManifest 丢弃)、坏 relation→边静默失效(R7)、缺 behavior→呆立、缺 dynamic→静态化。
- `reduce` / `decode` 外层包 try 边界:单条实体 decode 失败 → **丢弃该条**、不污染整帧(对齐 §1.4 越界整条丢弃);运行期坏边 throw → 标 `dead` + 跳过 + 诊断。
- **非对称兜底(E3)在分面语言里 = 逐分面降级**:表现侧宽兜底(降级显示、保留实例)、本质侧严兜底(非法即拒)。**隔离边界 = 分面边界。**

## 4. 定义 vs 实例分离(运行时只引用定义)

```js
// 运行时实例:只持 defId 一根指针 + 动态态。对齐阶段1 §1.4 内核 Entity。
Instance =        // 生物实例(进实体快照,§1.4 唯一权威 schema)
  {
    id,                       // entId(房主本地单调递增,迁移高位带 epoch 防撞号)
    m: {                      // manifest 段 = 对定义的引用(随快照传递与落盘,不可缺!)
      p,  // protoId(→ 锁定本质)        s,  // skin(资产引用,∈ allow.skins)
      a,  // pack(动作包 id,内核行为读这里,不读 proto)
      tr, // traits(⊆ allow.traits)     n,  // name(展示)      tu, // tune(钳窗后微调)
    },
    // 动态态(每 tick 由内核演进,非 manifest 一部分):
    x,y,z, ry, vx,vz,         // 位置/朝向/速度
    hp,                       // 当前血量
    st, stT,                  // 动作包状态机的状态编号 + 在此状态的 tick 计数
    state,                    // 物理状态系数来源(完好/受损/潮湿/灼烧?喂 toughness;经济 H1 待定,见 §8 经济 2.1)
    rng,                      // 确定性 RNG 子种子 = hash2(entId, spawnTick, world.seed)
  }

// 静态物实例 = 不存对象,只存方块 id(烘焙进 edits Map<"x,y,z", blockId>,阶段1 §2.3.1)。
//   静态物「实例化」= world.setBlock(x,y,z, bakedBlockId),零新协议。

// 迁移快照搬的是实例,不是定义:
MigrationSnapshot = {
  edits: [[x,y,z,id],...],   // 方块实例(种子+增量)
  ents:  Instance[],          // 生物实例快照(§1.4 线格式)
  // ⚠ 不含 DefinitionSet 本体!定义靠 REGISTRY_HASH(代码层)+ contentHash(可变层)双端同源。
}
```

> **工程意义**:迁移帧只搬实例(轻),定义不上线传(靠哈希同源)——这是 64KB 迁移墙能成立的根。`Instance.m` 段虽叫 manifest,但**只是一组指回定义的引用**(p/s/a/tr/tu),不是定义本体;成员收快照后用本地 `PROTOTYPES[p]` + `validateManifest` 复验(§3.4 防线的物理前提)。
>
> **lazy 重解析 = desync 结构性免疫**(总法则的运行期收益,见 §7 E6):实例只存能指,每次 reduce/渲染**当场用当前 DefinitionSet 解析**所指,从不缓存所指。定义一变,下次解析自动用新值——没有需要失效的缓存。**这是寻址模型自带的收益,不是补丁。**
>
> **两种解析时机(对静态物的诚实分叉,见 §8 阶段1 2.8)**:生物=运行期 lazy(持 m 段引用);静态物=发布期烘焙(实例只持 id,运行期不可重解析,改外观要改版本)。「所指总是运行期解析」对静态物走样——静态物的显现在烘焙时一次性定。
>
> **分叉判据 = `has(dynamic)`(§0.7 待拍点②裁决)**:有 dynamic 分面 → 活成 Instance(lazy 重解析);无 → 烘焙成 id(发布期定)。会动的方块 = 追加 behavior+dynamic 分面 → 自动转 Instance 侧。**注意 `Instance.m` 段必须留在迁移帧**(上一条「物理前提」)——这是成员独立复验的材料,绝不可为「瘦实例」抽走(否则反作弊从 A 档塌成「信房主」,见 §3.4 / 阶段1 §2.6)。

## 5. 具体实现(落到 VoxelCraft 栈)

> 引擎硬不变量与源码事实见事实底稿(已复核 world.js 215 行 / host.js 205 行 / constants.js 74 行 / textures.js / server/index.js / main.js)。本节只给数据结构、序列化预算、内存布局、双驱动、切片清单。

### 5.1 三个内核解析器 = 三组只读表(全是「能指→所指」查找)

能指是表的键(小整数下标或 32 字符哈希),所指是表的值(只读记录)。**解析 = 一次 Map/数组下标查找,永不重新计算。**

| 解析器 | 入参(能指) | 出参(所指) | 后备表(加载期 `Object.freeze` 深冻结) |
|---|---|---|---|
| 资产解析器 | assetId / packId | 纹理瓦片槽 / 动作包 FSM | `AssetPool`(Map)、`ACTION_PACKS`(数组) |
| 交互解析器 EffectResolver | actor/target + 目标 id 标签集 | 净效果 `effect`(正伤/零免/负疗) | `PROTOTYPES`(tier/maxHp/toughness 锁定令牌)+ 武器/护甲 `{标签→修正}` 数据包(contentHash)+ 形态系数常量 |
| 关系执行器 | 边 `{触发,gate,in,out}` | 生成事件 | 由对象定义的关系字段正/反查,无独立表(派生 FWD/BWD/DIST_BY_BIOME) |

注册表 = 有序数组(下标即稳定 id),加载期跑掉落上限静态断言(§1.3.1)。`REGISTRY_HASH = hashRegistry(PROTOTYPES, ACTION_PACKS, BUDGET_CLASSES, FACET_TYPES)`(★ `FACET_TYPES` = 分面类型集,§0.7 混合冻结:加分面类型即破封需评审)复刻 host.js `MAX_BLOCK_ID = Math.max(...Object.values(BLOCK))`(host.js:15)的「从数据派生、不能 desync」风格。

> **「概念是图、运行是表」(RDF 教训)**:实体—关系图是「定义期/构建期」的概念模型;运行期只保留烘焙后的扁平查找表 + 派生索引。**绝不在运行期内核实现通用图数据库**(违中端机内存)。图在编辑器里,数组在内核里。

### 5.2 DefinitionSet 序列化进世界状态 / 迁移(守 64KB + 2800 增量)

世界四段(阶段1 §0.1)= `种子 + 方块增量(≤2800) + 实体快照 + 资产清单`。DefinitionSet **不是第五段**,分裂落进既有两段:

- **静态物显现** → 烘焙成方块 id 进 edits(运行期房主自定义走 `64..255`,阶段1 §2.4),复用 world.js 既有 `[[x,y,z,id]]`(world.js:151-159)+ `Map<blockKey,id>` 去重(world.js:30),**零改协议**。
- **生物显现** → 实体快照(`m:{p,s,a,tr,n,tu}`+动态态),走 `ent.*` 消息族,不进 edits。
- **DefinitionSet 模板本身**(god 编辑的对象定义)→ 作为 `joined` 新可选字段透传(阶段1 §5.2「buildJoined 透传 rh/customBlocks/assetIds/ents」)。

**序列化形态(紧凑数组,能指=小引用,对齐实体快照 60-90B/条)**:

```js
joined.defs = {
  rh: <REGISTRY_HASH>,        // 比对不符 → 拒进房(阶段1 §2.7,复刻 bad-name toast)
  ch: <contentHash>,          // DefinitionSet 内容哈希(造物 §7.3 第二道闸门)
  add: [ /* [proto引用, skinAssetId, packId, traits位掩码, tune量化, 关系边引用...] */
         [9, 'a1b2…(32hex)', 1, 0b0010, 3, /*rel*/0,5], ... ],   // 房主新增定义
  del: [3, 7],                // 删除的内置定义下标(删除语义:已刷实例瞬死,造物 §6)
}
```

**预算量级(为什么能指必远低于上限)**:单条新增定义 ≈ 60-90B(与实体快照同量级,非巧合:都是「能指束」)。

| 段 | 条数上限 | 每条字节 | 段总量 | 占 64KB |
|---|---|---|---|---|
| edits(方块增量) | 2800 | 10-21 | 28-59 KB | **主占用** |
| defs(新增定义) | ~256 | 60-90 | ~15-23 KB | 吃 edits 预留 |
| assetIds(资产清单) | 192 | 16(二进制) | ~3 KB | 可忽略 |
| ents(实体快照) | 16(活体) | 60-90 | ~1.4 KB | 可忽略 |

> ⚠ **段总量为各自【独占帧】上限,非可同帧叠加。** 四段同时满载求和约 86KB(58.8+23+3+1.4)> 64KB,会越 `maxPayload=64*1024`、触 ws 1009 自动关帧(server/index.js:132 已核)。**硬约束:edits 接近 2800 满载时,defs 段必须走 `defsync` 分片帧、绝不与 edits 共 `joined` 帧;四段绝不可同帧满载。** 单帧实际只需装「当下要发的一段或两段轻段」,故工程上从不逼近 86KB。

**结论**:能指序列化本身无压力,但 **edits 才是 64KB 真正占用者**;**defs + edits 若同帧满载即破墙(见上「⚠」),必须靠 defsync 分片分帧才不违 64KB**。守恒:迁移帧 edits 已从 ~3000 下调到 ≤2800「为资产清单留预留」(阶段1 §4.2),defs 在 edits 未满载时可吃这预留共帧;一旦 edits 逼近 2800,把 defs 拆出独立 `defsync` 分片帧(复用 resync 多帧收敛,host.js:181)。成员收到后校 `rh`(注册表一致)+ `ch`(定义集一致)**两个哈希**。

### 5.3 资产内容寻址(搁置;重启上传/AI 管线时的落地口径)

完全沿用阶段1 §3.2(当前搁置,代码层素材占 `0..63`,字节通道休眠):

1. **强制降采样**:`createImageBitmap(png)` → 画到固定 **32×32 离屏 canvas**(显存安全阀 + 去重前提:同图无论原尺寸,降采样后字节一致——「先归一化再哈希」是去重前提,Git/IPFS 教训)。
2. **取 32×32 RGBA 字节**(16384B)做 `crypto.subtle.digest('SHA-256', bytes)` → **前 16 字节 hex = 32 字符 assetId**。
3. **去重 = Map 键碰撞**:`AssetPool: Map<assetId, {slot, bytes}>`;`has(id)` 则复用槽,否则分配下一空槽(64..255)。
4. **线格式** = 降采样后 PNG 字节(`toBlob`,0.5-2KB),分片 ws 传输(单片 ≤48KB)。assetId **绝不可改用自增**(造物 §7.1)。

**哈希碰撞裁决(可接受,不加防御)**:SHA-256 前 16 字节 = 128 bit。生日界 50% 碰撞需约 `2^64 ≈ 1.8×10^19` 份资产;单房间运行期自定义仅 192 槽、≤256 资产,跨房累积百万级碰撞概率仍 `<10^-26`。对私有、单房间百级资产的游戏,128 bit 远超需求。碰撞仅影响**表现层**(张冠李戴贴图),不影响本质/经济(本质走令牌不走哈希)——能指/所指分离的隔离收益。**前提:128bit 截断仅限单房间,不可当全局/跨房间永久库 id(那需全 32 字节)。**

**实现要点**:哈希是慢异步操作,**严格隔离在「上传/造物期」,绝不进 reduce/rAF/tick**。内核只见 assetId(已铸好的能指),从不碰字节。

### 5.4 实例态与定义的内存布局(≤256MB 堆、≤16 实体)

- **定义**:`PROTOTYPES`(冻结只读,全房共享一份)+ DefinitionSet(每世界一份,~256 条 × ~200B ≈ 50KB)。**实例绝不内嵌定义,只持 proto 下标(整数)。**
- **实例**:每实体对象 ~200-400B 堆(~10 数字字段 + manifest 整数下标 + 短字符串)。`16 × 400B ≈ 6.4KB`——**实体在 256MB 堆里完全可忽略**。真正占用方是区块(常驻 ≤1024 块 × 16384B ≈ 16MB)、纹理图集、three.js 几何缓冲(DESIGN.md 行 853-856:RADIUS 3/4、单 material、≤2 区块生成+≤2 mesh 重建/帧)。

**布局裁决**:实体存 `Map<entId, Instance>`(AoS)。**N≤16 时不做 SoA/archetype 列式存储**——cache locality 收益≈0(16 实体连一个 cache line 区间都填不满),而 archetype 的「增删组件搬数组 + 保持紧凑」是纯负债(ECS 反面教材)。AoS 可读、可 JSON 序列化直喂迁移快照、零迁移成本。SoA 留作重型服务器放开 1-2 数量级时的后路。**禁止引入任何 ECS 库**(违无构建 + 内核纯净)。

> **分面的 SoA 红线(§0.7 不可松)**:分面是**定义层纯数据可选字段**,加载期解析后**塌缩**进上述 AoS Instance / 烘焙表;运行期内核**绝不**持「按分面分桶」的列存(`Map<facetType, Array<态>>`)。`has(facet)` 派发 ≠ facet-bucketed storage——后者即 archetype,违本裁决。加载期可加断言:Instance 是单一 AoS 容器、无 per-facet 列数组。

### 5.5 性能(避免每帧分配、缓存解析结果、批处理)

- **reduce 零稳态分配**:解析结果是表查找不分配(`collectMods` 返回**预冻结**的 `{tag,val}` 数组、前缀匹配按标签 id 下标序累加,热路径不分配;无二维压制常量表);`out[]` 在 ctx 预分配每 tick `out.length=0` 清空;事件对象用对象池(`ent.spawn/move/hp/die/despawn` 五族各一可复用 shape)避免 GC 抖动。
- **WorldView 探针纯度陷阱**:`getBlock` 真只读可随便调(world.js:39-49);**`surfaceHeight` 非纯——内部 `ensureChunk` 拉起 16KB chunk**(world.js:162-175,行 167 已确证)。内核 ctx 只暴露真只读 getBlock,**禁止 reduce 里调 surfaceHeight**;第一切片把实体钳在房主已加载半径内(阶段1 §1.3)。这是底座「解析器纯函数」抽象容纳不了的副作用裂缝,处置 = 受约束探针 + 越界钳回断言(见 §8 阶段1 2.6)。
- **索引一次建长期用**:`FWD/BWD/DIST_BY_BIOME` 在 DefinitionSet 变更时重建一次,运行期只读。关系执行天然批处理(一次区块生成把该 biome 全候选一把取出按密度铺,而非逐实体反查)。
- **确定性 RNG**:`sd=hash2(entId,spawnTick,world.seed)` → `mulberry32(sd)`(禁 `Math.random`)。`rng` 进实体快照,迁移后续算一致。多输出按 dropPool 数组下标序逐个 roll(下标=稳定序,不依赖 Map 迭代序的不确定性)。

### 5.6 纯内核被 rAF 与 tick 双驱动(不碰 DOM/定时器)

**内核是纯函数,不持有时间源;驱动器在内核之外各自喂 dt。**

```js
// kernel.js(双端共享,禁 import three/DOM/定时器/rAF/WebSocket,禁 Math.random/Date.now)
export function reduce(state, inputs, dtWall, ctx) {
  let acc = state.acc + dtWall, ticks = 0;
  const out = ctx.out; out.length = 0;
  while (acc >= DT_FIXED && ticks < MAX_CATCHUP_TICKS) {  // DT_FIXED=1/10s;封顶防螺旋
    stepFixed(state, inputs, DT_FIXED, ctx, out);
    acc -= DT_FIXED; ticks++;
  }
  state.acc = acc;
  return { state, out };
}
```

- **浏览器(轻量)**:`sim-driver.js` 挂 main.js 既有 rAF loop(main.js:1064 `requestAnimationFrame(loop)`、1066 `dt=(now-lastTime)/1000`、1069 `if(dt>0.05)dt=0.05`)。dt 钳 0.05 天然实现「后台冻结 + 回前台轻补封顶」(阶段1 §1.5)。**host.js 仍零定时器**(DESIGN.md 行 715-717),时间唯一源是 main.js rAF。
- **服务器(重型)**:`server-driver.js` 用 `setInterval` + `process.hrtime` 喂同一 `reduce`(DESIGN.md 行 304「no-timers rule binds host.js only, not the server」;server/index.js:315-322 心跳已有 setInterval 先例)。可放开 `MAX_CATCHUP_TICKS` 与实体数 1-2 数量级。**server/index.js 仍哑中继零游戏逻辑**。

两端喂同一纯 `reduce`、同一 `PROTOTYPES`/`ACTION_PACKS`(同 REGISTRY_HASH)、同一确定性 RNG → 模拟逐位一致。DOM/three/定时器全在驱动器层,被 import 边界挡在内核外。

## 6. 既有经验与坑(prior art:采纳 / 避免)

本底座是 **RDF 三元组 + 内容寻址 + Minecraft registry + 符号驻留** 四者的工程合流。九类成熟模式无一要求改变已锁定模型,相反逐条**背书**它。落地风险集中在两点:**(1) 别把概念上的图/ECS/RDF/通用注册表原样实现进运行期内核**(违中端机内存 + 帧宽 + 纯净);**(2) 别越过「数据非代码」红线**(别抄 datapack function、别让配方跑表达式),否则 REGISTRY_HASH 防线作废。

| 模式 | 角色 | 采纳 | 避坑红线 |
|---|---|---|---|
| **Minecraft** registry/datapack/loot/recipe/tags | 直接祖师爷 | 双 id(哈希=防冲突 / 下标=快寻址);loot/recipe/tag 全做数据;**tag 设第一公民**承载分布与采集偏置 | 不抄 `function` 命令脚本(数据非代码红线);不在帧里下发注册表(只下发 REGISTRY_HASH);tag 合并须确定性序(否则 contentHash desync) |
