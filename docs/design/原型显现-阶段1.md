## 0. 摘要与本阶段范围

### 0.1 一句话定位

把《山根乡》的「对象本体论 + 双系统运行底座」打到能跑通的最小地基：**世界 = 种子 + 方块增量 +（新增）实体快照 +（新增）资产清单**，四段都靠「人人持本地副本 + 房主权威 + 迁移收敛」存活；**一份纯模拟内核**两端共用，浏览器用 rAF 喂、服务器用 setInterval 喂；**显现只能从开发者做好的有限库里重组**，经纯函数预算校验器签发。

### 0.2 第一切片要验证什么（唯一目标）

1. **静态造物台**：玩家放一件确定性、无 AI 的静态物（方块/植物/装饰/家具/房屋组件）→ 烘焙成方块 id → 进现有 `edits` 增量通道 → 零破坏 `block`/`joined`/`resync`。
2. **一只会逃跑的木级猎物**：极小纯内核由 `main.js` 的 rAF loop 每帧喂一次 tick → 产出实体事件 → 房主 cast → 成员镜像渲染。
3. **实体快照随迁移传递**：房主退出 → 新房主 `resync` 带实体快照 → 全员收敛（承认轻量模式的弱保真）。
4. 全程落在中端机包络内（同屏 ≤16 实体、≤256MB 堆工作集、≥30fps）。

### 0.3 本阶段刻意不做（留后续切片 / 留重型模式）

- 生物的附性/动作包多样性（第一切片只 1 个 `flee_skittish` 包）。
- tier 1+ 内容、制造系统、tune 微调的复杂窗口（静态物 tune 为空）。
- 自定义贴图的成员→成员回收（v2 优化）、举报机制、255 槽耗尽后的分片协议。
- 重型模式服务进程的完整实现（本文只定接口与分工，不实现）。
- **权威经济/背包的反作弊根治**——只有重型模式服务器能做（见 §2.6 诚实边界）。

### 0.4 现有引擎硬不变量对账总表（设计绝不破坏）

| 不变量 | 本设计的姿态 |
|---|---|
| `server/index.js` 哑中继：零游戏逻辑、只转发不透明字节、单帧 64KB、迁移提升最老成员 | **零改动**。实体事件、资产分片都是普通不透明 `d` 载荷 |
| 世界 = 种子 + 方块增量（`Map<"x,y,z", id>`，去重，线格式 `[[x,y,z,id]]`） | 前两段一字不改；实体与资产是**正交的新通道**，绝不混入 edits 数组 |
| `host.js` 零定时器、纯事件驱动 | 内核也零定时器；时间唯一源是前台 `main.js:1062` rAF loop（见 §1.5 冻结语义） |
| 迁移不传世界，靠人人持全量本地态 + `resync` 收敛 | 实体快照、资产清单都遵循同构「房主态胜出」 |
| 方块 id 0..8，`MAX_BLOCK_ID=8`，双端不一致扩展即静默丢弃 | id 空间统一分区（§2.4），落地校验必须三处收敛同一谓词 |
| 无外网资源、纯 ES modules 无构建 | 所有新模块是可直接 import 的 ESM；自定义贴图全 P2P over ws，零 CDN |
| 中端机 ≤256MB 堆 / 30fps / 纹理设硬阀 | 所有上限按此反推（§4） |

## 1. 双系统与纯模拟内核

### 1.1 双系统分工（已锁定，本节只落实）

| | 轻量模式 | 重型模式 |
|---|---|---|
| 房主 | 玩家浏览器标签页，可迁移 | 云服务器，永远在线 |
| 时间源 | 前台 `main.js` 的 rAF loop（仅前台，后台冻结） | `setInterval(1000/TICK_HZ)` + `process.hrtime`（永不冻结） |
| 世界落盘 | 不落盘，靠迁移时人人持本地态 | 落盘到磁盘，不靠迁移 |
| 64KB 帧约束 | **受约束**（迁移帧硬墙，见 §4.2） | **不受约束**（不迁移、服务器无帧上限） |
| 反作弊 | 只能「复验 + 丢弃」，房主即潜在攻击者（§2.6 B） | 服务器是唯一可信计算基，根治（§2.6 C） |
| 实体上限 | ≤16 只活体（中端机护栏） | 可放开 1–2 个数量级 |

**核心：战斗/怪物/制造/原型结算写成一份纯模拟内核**——状态进 + 事件进 → 新状态出；不碰 DOM、不自带定时器。两端复用同一份 `kernel.js`，唯一区别是谁算 `dtWall`。

### 1.2 纯模拟内核接口（零 DOM / 零定时器 / 两端同源）

新增纯 ESM 模块（顶部用禁词注释钉死纪律，可被 grep lint 守住）：

| 模块 | 职责 |
|---|---|
| `public/js/sim/kernel.js` | 唯一入口 `reduce(state, inputs, dtWall, ctx) -> {state, out[]}`，纯函数 |
| `public/js/sim/protos.js` | 原型表（冻结），等价 §2.1 的 `PROTOTYPES` |
| `public/js/sim/actionpacks.js` | 动作包库（冻结），等价 §2.5 |
| `public/js/sim/rng.js` | 确定性 PRNG（复刻 `mulberry32`/`hash2`，不用 `Math.random`） |
| `public/js/sim/snapshot.js` | 快照编解码（**唯一权威实体快照定义**，见 §1.4） |
| `public/js/manifest-validator.js` | 纯函数预算校验器（§2.5），双端同源 |

**禁止 import**：three、DOM、window、`setInterval/Timeout`、`requestAnimationFrame`、WebSocket、`host.js`、`main.js`、`renderer.js`。**禁用** `Math.random` 与 `Date.now/performance.now`——随机只走 state 内携带的 seeded rng，时间只来自外部喂入的 `dtWall`。

**关键决定：内核状态不含方块体素**。方块仍归 `World`（种子 + 增量）。内核通过只读探针 `WorldView` 查方块、绝不写。这让内核可在中端机 ≤256MB 预算内只背 ≤数十实体，不背 16KB/区块的体素。

```js
// WorldView 契约（浏览器侧直接传 world，服务器侧实现同名两方法）
WorldView = {
  getBlock(x,y,z),        // 真只读：未生成区块返回 AIR（world.js:39 已核实，安全）
  surfaceHeight(x,z),     // ⚠ 非纯只读，见 §1.3
}
```

`reduce` 内部用**固定步长 `DT_FIXED = 1/10s`** 把墙钟 dt 切成整数个确定性 tick（不足进 `acc` 累加），一举解决：① 浏览器 30fps 与服务器步长差；② 后台→前台补算；③ 确定性。带 `MAX_CATCHUP_TICKS` 封顶防卡死后一次补上万 tick。

- **输入事件（意图）**：`spawn` / `attack` / `scare` / `remove`。
- **输出事件（已结算事实）**：`ent-spawn` / `ent-move` / `ent-hurt` / `ent-remove`。

### 1.3 ⚠ `surfaceHeight` 不是纯只读（吸收审查发现，修正预算论证）

已核实 `world.js:162` 的 `surfaceHeight()` 内部调 `ensureChunk()`（`world.js:167`），会生成并永久保留一个 16KB 的 chunk `Uint8Array` 进 `world.chunks`。逃跑猎物正是会跑远的——若内核每 tick 对实体位置调 `surfaceHeight(floor(e.x), floor(e.z))`，只读探针会**静默触发未生成区块的体素生成与常驻**，既违背「探针只读」，也给「内核不背体素、≤256MB」开隐性缺口。

**定案（择一，落地时执行）**：

1. **首选**：`WorldView.surfaceHeight` 在契约里**显式标注「可能触发区块生成、非纯只读」**，并据此修正预算：内核背的不只是实体，还间接拉起实体足迹覆盖的区块。但因第一切片同屏 ≤16 实体、且实体只在房主已加载半径（RADIUS=3，49 区块）内活动，足迹区块本就在 `world.chunks` 里，**不额外拉起新区块**——故预算仍成立，但论证依据从「探针不碰体素」收紧为「实体活动范围 ⊆ 已加载半径」。
2. **备选**：内核改用 `getBlock` 自下而上探地（真只读，未生成区块返回 AIR → 实体落到 y=0 默认地面，可接受），完全不调 `surfaceHeight`。

**第一切片采用方案 1 + 硬约束「实体不得游走出房主加载半径」**：内核 `stepEntity` 若实体将越出 RADIUS×CHUNK 范围，把它钳回边界（逃跑猎物本就该在玩家附近），从根上杜绝探针拉起新区块。这条写进与 `world.js` 现状对账的自检表（§6）。

### 1.4 实体快照：单一权威 schema（吸收 blocker #1）

> **审查发现的最核心断裂**：原型—显现 §2.3 要求每个生物快照携带**完整 manifest**（skin/pack/traits/name/tune + 子种子），因为显现层全部可变性都在这里，且成员要靠它复跑 `validateManifest` 防伪；但纯模拟内核原文的快照元组只有 `[entId, protoIdx, x,y,z, ry, hp, stIdx, seedLo]`，**skin/pack/traits/name/tune 全部缺失**。后果：迁移/落盘后「蓝羽」退回裸 proto 默认显现；成员收不到 skin/pack/traits 无法复验，§2.6 的防线落空；内核「proto 指固定 action」与显现层「pack 可在 allow.actionPacks 里挑」互斥。
>
> **本文裁决：实体快照只有一份定义，内核 Entity 与线格式都带 manifest 段。** 内核行为读 `entity.m.a`（pack），不读 proto 上的固定 action。

#### 1.4.1 运行时主存 Entity（内核状态的一部分）

```js
// 内核状态：Map<entId, Entity>。entId = 房主本地单调递增整数（迁移时高位带 epoch 防撞号）。
Entity = {
  id,                 // entId
  m: {                // manifest（显现层全部可变引用，随快照传递与落盘）—— 不可缺！
    p,                // proto（protoId，指向锁定原型，决定阶级/掉落/采集/战斗定位）
    s,                // skin（贴图/资产 id，∈ allow.skins）
    a,                // pack（动作包 id，∈ allow.actionPacks）—— 内核行为读这里，不读 proto
    tr,               // traits（附性 id 列表，⊆ allow.traits）
    n,                // name（1..16 码点，cleanName，仅展示）
    tu,               // tune（微调，钳到预算窗，见 §2.5）
  },
  // ---- 动态态（每 tick 由内核演进，不是 manifest 的一部分）----
  x, y, z, ry,        // 位置 + 朝向
  vx, vz,             // 速度
  hp,                 // 当前血量
  st, stT,            // 动作包内状态机的状态编号 + 在此状态的 tick 计数
  rng,                // 该实体的确定性 RNG 子种子（保两端/续跑一致）
}
```

#### 1.4.2 快照线格式（`sim/snapshot.js` 唯一定义，两文档交叉引用此处）

定长 + 变长两段拼接，省 JSON key：

```js
// 单实体编码（紧凑数组，非对象）：
[ id, p, s, a, hp, st,                       // 6 个小整数：身份 + 显现引用 + 动态
  Math.round(x*100), Math.round(y*100), Math.round(z*100), Math.round(ry*1000), // 定点
  rngLo,                                      // RNG 子种子低位
  tr,                                         // 附性数组（短，[]或[0,1]）
  n,                                          // name 字符串（≤16 码点）
  tu ]                                        // tune 对象（第一切片为空 {}）
```

- **字节预算（同事实基线 A4 口径）**：单实体含 manifest ≈ 60–90B；1 只 ~90B、16 只 ~1.5KB、50 只 ~4.5KB、200 只 ~14KB——**远不触 64KB**。真正贴 64KB 的是 edits（A4 既有约束），实体快照独立、不加重它。
- **信封**：`{sv, tick, rng, next, e:[...]}`（sv=schema 版本、tick=内核 tick、rng=全局 rng 态、next=下个 entId）。
- **`decodeEntSnapshot` 严格校验**：用与 `main.js:739` edits 过滤同等严格度——每个数值字段 `Number.isInteger`/有限、`p` 落在 `PROTO_COUNT` 内、越界 proto 整条丢弃；`n` 过 `cleanName`；并对解出的 manifest **跑一遍 `validateManifest`**（§2.5），失败整条丢弃。**这就是 §2.6 防线成立的物理前提：快照里带齐 skin/pack/traits/tune，成员才复验得了。**
- **`protoIdx`/`pack`/`skin` 双端必须一致扩展库才能跨端落地**（等价 `MAX_BLOCK_ID` 纪律的实体版，靠 §2.7 的 `REGISTRY_HASH` 闸门兜底）。

### 1.5 后台冻结语义（与无定时器/迁移共存）

**定案：房主切后台 → 内核时间冻结（不补算），回前台轻补封顶丢弃积压**（怪物像打了个盹，不重演几小时）。

理由：`host.js` 零定时器，内核零定时器，时间唯一源是 `main.js:1062` 的 rAF loop。后台 rAF 被浏览器节流 → `reduce` 不被调用 → `state.tick` 不前进 → **怪物天然冻结，这是落在现有架构真空里、不是 bug**。`loop` 已有 `dt>0.05` 钳制（`main.js:1069`，已核实）天然实现「冻结 + 轻补」。触屏房主已有 `acquireWakeLock()`（`main.js:888`）减少切后台概率，但息屏仍冻结——**这是已接受的轻量模式天花板，重型模式才有永续真 tick**。

### 1.6 木级会逃跑猎物最小走查

- **每 tick 行为**：动作包 `flee_skittish.step`（纯函数）——感知半径内有玩家/scare 源则进 flee，背离方向 + 确定性抖动（实体局部 rng）逃跑（speed 3.2 < 玩家 5.5，逃得掉追得上），只读 world 探针做碰撞、贴地（受 §1.3 约束）、calmTicks 后回 idle。
- **attack 结算带非零下限**（对齐已锁定决定 #3）：class 差 ≥3 仍夹至 ~1% 伤害，允许低级屠龙的风味。
- **同步给成员（定案：成员不跑内核）**：权威只在房主跑，成员靠房主广播 `ent` 系列消息（见 §2.3）镜像渲染，沿用 `pmove` 的「房主权威回声」心智模型，零新概念，省成员预算。`move` 用现有 `MOVE_INTERVAL`=12Hz 节流 + 覆盖语义。成员侧入站校验同 §1.4.2 严格度。
- **迁移续命**：成员本地维护「影子镜像」（只写镜像不 reduce），被提升时影子升为权威内核态（epoch 高位防撞号）+ `resync` 带快照广播；其余成员用新房主快照全量覆盖本地影子（幂等）——与「房主态胜出」（DESIGN.md:204-205）完全同构，只是对象从方块变实体。收 `joined` 前被提升的边界沿用 `main.js:842-848` 的 `net.close()`。

## 2. 原型—显现数据格式与平衡执行

### 2.1 四层本体论（已锁定决定 #2，本节只落实）

**「显现无限」= 对一个有限的、开发者预先做好并调过平衡的资产/动作包库的无限重组。** 因此交付的不是「让玩家编 AI 的格式」，而是「一个只能引用库里编号的格式 + 一个纯函数预算校验器」。

| 层 | 谁写 | 可变性 | 落地 |
|---|---|---|---|
| **原型注册表** | 开发者（随版本发布的只读 JS 常量） | 版本内不可变，玩家不可碰 | `prototypes.js`（双端 import） |
| **资产/动作包库** | 开发者（只读常量） | 版本内不可变 | `actionpacks.js` / `assets.js` |
| **显现** | 玩家/造物台产出，经校验器签发 | 可变，但每字段只能引用上两层编号 | 静态物→增量；生物→实体快照 |
| **预算校验器** | 开发者（纯函数） | 不可变 | `manifest-validator.js`（双端同源） |

### 2.2 原型注册表 Schema（锁定阶级/用途/掉落池/生态位/采集门槛/战斗定位）

原型注册表是**有序数组，下标即 `protoId`（稳定整数，永不复用，弃用只置 `deprecated:true`）**。这刻意复刻现有引擎成熟模式（`BLOCK`/`BODIES`/`PALETTE` 全是「下标即稳定 id」），使版本一致性校验能像 `MAX_BLOCK_ID` 一样从表派生。

```js
// prototypes.js — 开发者发布的只读注册表，双端 import。全字段 LOCKED。
export const PROTOTYPES = [
  {
    protoId: 0, key: 'fowl_small', kind: 'creature',
    tier: 0,                       // 0=木 1=石 2=铁 3=黄金 4=钻石。压制公式唯一输入，显现不可改
    role: 'prey', biome: ['grass','forest'],
    harvest: { method: 'hunt', minTier: 0 },
    dropPool: [                    // 锁定 —— 「不能掉钻石」的物理写死处，显现层无对应可写字段
      { item: 12, chance: 1.0, min: 1, max: 2 },
      { item: 13, chance: 0.5, min: 1, max: 1 },
    ],
    combat: { maxHp: 8, baseAtk: 1, baseDef: 0, disposition: 'flee' },
    allow: {                       // 白名单（不是黑名单）：显现只能从这里挑
      actionPacks: [0,1,2], skins: [0,1,2,3,4], traits: [0,1],
      budgetClass: 'creature_t0',
    },
    deprecated: false,
  },
  // ... protoId 1, 2, 3 ...
];
export const PROTO_COUNT = PROTOTYPES.length;
export const REGISTRY_HASH = /* §2.7：对锁定字段的确定性哈希 */;
```

**关键不变量**：
- `dropPool` 是锁定字段，显现 schema **没有任何能触碰它的键**——这是「不能掉钻石」最根本的防线。
- `allow.*` 是白名单。显现引用不在白名单里的动作包/贴图 → 校验器直接拒签。
- `tier` 是压制公式唯一输入，显现不可改。
- **掉落上限不变式（发布期 + 运行期双查）**：对任一原型，`dropPool` 内每个 item 的 `PROTOTYPES[item].tier` ≤ 本原型 `tier`。加载注册表时静态断言，违规则该版本客户端启动即报错（开发期就拦住）。制造路线另有「钢=铁造但封顶黄金、不可递归升钻石」的同类上限，在制造系统校验，本章只立掉落侧。

### 2.3 显现进世界状态：两条互斥通道

#### 2.3.1 静态物 → 进「种子 + 增量」（烘焙成方块 id）

静态物 = 方块/植物/装饰/家具组件——确定性、无 AI、无自走时间。**必须落进现有 `edits` 增量通道**（那是迁移成立的根）。

**设计：静态物显现在发布期就被预解析为一组方块 id，登记进 `BLOCK` 表。** 世界状态里不存显现对象，只存方块 id——和现在的 `LOG`/`PLANK` 完全一样。

```text
显现（开发者发布期）──预烘焙──▶ 新增 BLOCK 条目（id, FACE_TILES 映射）──▶ 双端 constants.js/textures.js 一致扩展
玩家放置（运行期）──▶ world.setBlock(x,y,z,id) ──▶ 进 edits Map ──▶ 走现有 block/joined/resync，零新协议
```

**为什么不把显现对象塞进 edit**：现有 `block` 每条只 10–21 字节（事实基线 A4），一帧 64KB 装 ~3000–6000 条。若每条改塞 JSON 显现对象（几十~上百字节），64KB 容量腰斩到几百条，大型造物迁移立刻撞墙。**烘焙成 id 保持每条增量个位数字节，完全复用已验证的容量包络；迁移 `resync` 只搬 `[[x,y,z,id]]`，无需理解显现，迁移路径零改动。**

#### 2.3.2 生物 → 进「实体快照」（不进 edits）

生物有自走时间，**绝不能进 edits**（增量是静态确定性数据，且 `host.js` 无 tick 源）。生物进**实体快照**（§1.4 的唯一权威 schema），与方块增量正交。

房主 → 成员的实体增量广播（新消息族，走现有 `net.cast`，遵循「房主权威 + cast」信任模型）：

```js
{ t:'ent', op:'spawn',  e, m:{p,s,a,tr,n,tu}, x,y,z,ry, hp, st }  // 生成：含完整 manifest（仅此一次）
{ t:'ent', op:'move',   e, x,y,z,ry, st }                          // 高频：仅动态态，≤12Hz 节流
{ t:'ent', op:'hp',     e, hp }                                    // 受击
{ t:'ent', op:'die',    e, drops:[[itemProtoId,count],...] }       // 死亡 + 掉落（房主裁定）
{ t:'ent', op:'despawn',e }
```

**体积控制**：`spawn` 含完整 manifest（短键，~60–90B），只发一次；此后只发 `move`（~30–40B，与 `pmove` 同量级）。同屏活跃实体硬上限 ≤16（§4.2），全量快照一帧 64KB 轻松容纳。

#### 2.3.3 迁移时实体快照怎么传（诚实：保真度低于静态世界）

成员只持有「房主广播给它的实体投影」，不是权威态，不满足「人人持完整权威副本」前提。**定案：房主态胜出 + 快照重广播，与 edits 的 `resync` 同构，但语义更弱。**

```js
{ t:'resync', edits:[...], ents:[<新房主本地持有的全部实体快照>...] }  // 见 §4.2 合帧预算
```

- 新房主用死房主窗口前最后收到的实体投影作为权威基线。死窗口内的实体演进（移动/死亡）若新房主没收到 → 丢弃，以新房主投影为准。逃跑的鸡可能「瞬移回几百毫秒前的位置」——**可接受的迁移瑕疵**，与现有 edits 迁移的「死窗口编辑丢失」一致。
- 实体 RNG 子种子随快照传递，使新房主接管后演进仍确定性。
- ⚠ **轻量模式固有弱点：实体态迁移保真度低于静态世界。** 重型模式服务器永远在线、世界落盘、不靠迁移，此问题不存在。

### 2.4 方块 id 空间统一分区（吸收 blocker #3 + minor #1）

> **审查发现的冲突**：原型—显现原文写「静态物可用 id 9..255」并举例占 id=9；自定义 PNG 原文写「内置 0..63 保留，自定义从 64 起」。同一 id（9..63）在两份里归属不同，先撞自己人。

**本文裁决：唯一 id 分区表，两文档引用此处。**

| id 区间 | 归属 | 谁分配 | 数量 |
|---|---|---|---|
| `0..8` | 现有内置方块 | 已发布 | 9 |
| `9..63` | 开发者发布的**静态造物方块**（§2.3.1 烘焙） | 随客户端版本发布 | 55 |
| `64..255` | 运行时房主分配的**玩家自定义贴图方块**（§3） | 运行期房主 | 192 |

> **⚠ 落地校验关口（吸收 minor #1，钉死三处口径必须收敛为同一谓词）**：
> 已核实**当前三处对 id 上界的口径本就不一致**：
> - `world.js:19`（`World.load` 存档过滤）已是 `e[3]<0 || e[3]>255`（即已放行 0..255）；
> - `host.js:15` 的 `MAX_BLOCK_ID = Math.max(...Object.values(BLOCK)) = 8`；
> - `main.js:122` 同样 `MAX_BLOCK_ID=8`，运行期 `block`（`main.js:777`）/`joined`（`main.js:741`）/`resync`（`main.js:798`）校验都是 `id <= MAX_BLOCK_ID`（=8），`host.js:110` 入站 `block` 同样 `id <= MAX_BLOCK_ID`。
>
> 因此落地时**绝不能只抬高一个常量**。三处必须收敛为**同一谓词**：
> ```js
> isKnownBlockId(id, customBlocks) =
>   Number.isInteger(id) && id >= 0 &&
>   (id < 64 ? id <= STATIC_BLOCK_MAX   // 内置 0..8 + 已发布静态 9..63
>            : customBlocks.has(id));   // 自定义 64..255：必须已在 customBlocks 清单声明
> ```
> 若三处放宽方式不一致（例如运行期简单放宽为 `≤255` 而非 `∪ customBlocks 已声明`），就会重新打开事实基线 A3 描述的缺口：**要么未声明的高 id 被静默丢弃，要么未声明 id 被放进 edit 污染世界与存档**。`world.js` 现状已是 0..255、与 `host.js`/`main.js` 的 `≤8` 不一致——放宽时必须三处一起改到同一谓词。本文标注此为必经关口，**不在此实施**。

### 2.5 平衡执行：动作包库 + 预算校验器

#### 2.5.1 动作包库（不是脚本，不是可执行代码）

动作包 = 开发者预先做好、调过平衡的**有限状态机定义 + 参数**。玩家显现只能引用 `packId`，不能定义新包——这是「不允许玩家自由编 AI」的强制点。

```js
// actionpacks.js — 只读库，packId = 数组下标。
export const ACTION_PACKS = [
  {
    packId: 1, key: 'skittish_flee',
    states: [                       // 内核认识的固定「行为原语」编号，原语实现写死在内核里
      { id: 0, name: 'idle', prim: 'wander', speed: 1.0, sense: 6 },
      { id: 1, name: 'flee', prim: 'flee',   speed: 3.0, sense: 10, from: 'player' },
    ],
    transitions: [
      { from: 0, to: 1, when: 'threat_within', d: 8 },
      { from: 1, to: 0, when: 'threat_beyond', d: 14 },
    ],
    budget: { aggression: 0, mobility: 3, sense: 10, complexity: 2 },  // 开发者调平衡时填的强度指纹
  },
];
```

内核只认固定 `prim` 集合（`wander/flee/chase/idle/...`），每个原语是内核里写死的纯函数；包只能参数化组合，不能引入新原语。库随客户端版本走，零网络传输。

#### 2.5.2 预算校验器（纯函数，四维裁决）

`validateManifest(manifest, registry, library) -> { ok, reason } | normalizedManifest`。无 DOM、无定时器、无网络，双端同源、幂等、确定性（同输入恒同输出，使双端能独立得出同一裁决）。

| 预算维度 | 校验内容 | 拒签示例 |
|---|---|---|
| **强度** Power | `tune` 微调后的 hp/atk/def 落在 `budgetClass` 窗口内 | 木级 `tune.hp:+50` → 超窗 |
| **经济** Economy | 引用的 skin/pack/traits 全在 `allow.*`；掉落池不可被显现触碰（根本无此字段） | `pack` 不在 `allow.actionPacks` |
| **行为** Behavior | `ACTION_PACKS[pack].budget` 各维 ≤ `budgetClass` 上限 | 木级 prey 引用 aggression>0 的捕食者包 |
| **表现** Presentation | skin 引用合法、traits 数 ≤ 上限、name 过 `cleanName`（1..16 码点） | traits 超长 / name 非法 → 截断或拒 |

```js
export const BUDGET_CLASSES = {
  creature_t0: {
    power:  { hp:[6,12], atk:[0,2], def:[0,1] },
    behavior: { aggression:0, mobility:3, sense:12, complexity:3 },
    presentation: { maxTraits:2 },
    tuneKeys: ['hp'],            // tune 只允许出现这些键
  },
  // creature_t1 ... t4 逐级放宽
};
```

**输出契约**：`ok:false` → 不进世界状态（造物台拒绝产出 / 内核拒绝 spawn）；`ok:true` → 返回规范化 manifest（裁剪非法键、name 过 cleanName、tune 钳到窗口边界而非直接拒，更友好）。

#### 2.5.3 确定性 RNG（让双端能独立复算）

每个实体生成时分配子种子 `sd = hash2(entId, spawnTick, world.seed)`。掉落判定走 `mulberry32(sd_drop)` 序列消费，**不是 `Math.random()`**。任何成员可用同 `sd` 复算房主声称的掉落，对不上即丢弃。这把「裁决可验证性」从「信任房主」降级为「房主必须出示可复算的随机源」。

### 2.6 轻量模式可信度的诚实边界（最关键的诚实交代）

前提：**轻量模式房主 = 某玩家的浏览器标签，运行权威 `HostRoom`，可被改源码/改内存。** 现有引擎已有部分免疫（`host.js`/`main.js` 双向校验、`adoptMembers` 重跑 cleanName/cleanSkin 防旧房主投毒）。在此基础上分层：

**A. 成员侧能强制的（靠人人持同一份只读库 + 纯校验器，无需服务器）**

| 攻击 | 防线 | 为什么成立 |
|---|---|---|
| 房主下发引用超白名单的 skin/pack/traits | 成员对 `ent.spawn` 跑**同一个 `validateManifest`**，失败丢弃 | 库与校验器双端同源只读；快照带齐 manifest（§1.4）才复验得了 |
| 房主下发超出 id 谓词的方块 id | 现有引擎已强制静默丢弃（§2.4 谓词） | 既有不变量，白嫖 |
| 房主篡改原型/库（改 `prototypes.js` 让木鸡掉钻石） | `REGISTRY_HASH` 不匹配 → 拒绝整个 `joined`/`ent`（§2.7） | 篡改即改哈希 = 版本不一致，本就该拒 |
| 房主给实体灌超窗强度（hp 9999） | 成员侧 `validateManifest` 强度预算把 tune 钳到窗口 / 拒 spawn | 窗口在只读 `BUDGET_CLASSES`，房主改不动成员手里的副本 |

**B. 成员只能「检测 + 不信任」、无法「阻止房主自己得益」（轻量模式真实软肋）**

| 攻击 | 现实 | 缓解（非根治） |
|---|---|---|
| 房主声称「木鸡掉了钻石」 | 成员可用 `sd` 复算 dropPool 发现不符 → 丢弃这条 drop；但房主在自己本地仍可把钻石写进背包，成员管不着 | 成员对 `die.drops` 跑「掉落上限规则 + sd 复算」，不符则不渲染、不计入自己拾取；举报机制留作开放问题 |
| 房主让实体瞬移/穿墙 | 成员能看出 `move` 不符物理，但轻量模式没有第二个权威推翻房主 | 成员侧物理 sanity check（速度上限 = 包 speed × 阶级系数），越界则钳制显示；改不动房主本地真相 |
| 房主无中生有 spawn 合法但本不该刷的高价值实体 | manifest 合法（过校验），成员无法证明「它本不该出现」 | 轻量模式无解——刷怪规则也在房主手里 |

**根因（必须写明）**：轻量模式**没有独立第三方裁判**。房主既是规则执行者又是潜在攻击者。成员的极限是「复验一切可复验的、丢弃一切对不上的」，但**房主对自己本地背包/世界的写入，成员物理上无法否决**——迁移模型的根就是「房主态胜出」。

**C. 只能靠重型模式服务器根治的**

| 性质 | 为什么必须服务器 |
|---|---|
| 权威背包/经济的最终真相 | 服务器是唯一权威，玩家客户端无任何权威态，只发输入收结果 |
| 刷怪/掉落不可伪造 | RNG 种子由服务器持有，玩家无法预测/伪造 |
| 跨房间/持久经济 | 世界落盘在服务器，不靠迁移，无弱一致语义 |
| 反作弊硬保证 | 服务器 = 不可被任一玩家篡改的可信计算基 |

**结论（对齐用户锁定决定）**：轻量模式把「木级蓝鸡不能掉钻石」**在数据层做到了「房主无法让诚实成员相信钻石掉落」**（dropPool 锁定 + 双端复验 + sd 复算 + REGISTRY_HASH），但**做不到「阻止作弊房主自己在本地拿到钻石」**。后者**只有重型模式服务器能根治**。同一份纯内核 + 同一校验器在两种模式跑相同逻辑，差别纯粹是「裁判可信度」——这正是双系统设计的意义。

### 2.7 REGISTRY_HASH：版本一致性闸门

```js
// prototypes.js / actionpacks.js 各导出对「锁定字段」的确定性哈希（复刻 hash2 整数混合风格，纯函数）
export const REGISTRY_HASH = hashRegistry(PROTOTYPES, ACTION_PACKS, BUDGET_CLASSES);
```

- `joined` 扩展携带 `rh:<REGISTRY_HASH>`。成员比对本地值，不匹配 → 拒绝进入该房间（toast「版本不一致」），复刻现有 `bad-name`/`full` 风格。
- 这把「房主篡改注册表」从「能注毒」变成「连不进去」。
- ⚠ 要求所有客户端跑同一版本。私有、不公开、LAN 联机场景下这是合理假设（玩家从同一处拿同一份 `dist`）。版本升级需所有人同步——已知运维约束，非缺陷。

## 3. 自定义 PNG 贴图存储方案（内容寻址分片 + 房主资产清单）

> ⚠ **本章方案已被《造物与制造体系》搁置**：现行模型为**素材走代码层、随构建发布、定义只存引用**（见 [`造物与制造体系.md`](造物与制造体系.md) §3.4 连带简化 与 §8 搁置项）。重启条件 = 做**游戏内上传 / AI 管线**（接入点 [`造物与制造体系.md`](造物与制造体系.md) §7.1 AssetSource，id 须为内容哈希）。**本章保留作为重启时的落地参考，第一切片不实施。** 下文 §4.7/§4.8/§5.1（`assets.js`）与 §6.1 开放问题 #4 同此搁置，各处另有回指。

> 本章只解决：玩家自带的 PNG 怎么进游戏、怎么发给同房间所有人（含后加入者和迁移后新房主）、怎么不撑爆中端机显存。三条硬约束：① 中继单帧 64KB；② 局域网可能无外网，禁 CDN；③ three.js r184 内置、纯 ESM 无构建。

### 3.0 三个被直接命中的源码事实

1. **中继是只读静态服务器 + 哑中继，没有也不该有上传写路径。** `server/index.js` HTTP handler 只接受 GET/HEAD（其余 405），ws 层只转发不透明 `d`。任何「在中继加上传路由」的方案都要改服务器、引入磁盘写、破坏「哑中继零状态」——直接否掉。
2. **贴图系统是「一张 canvas 图集 + 一个共享材质」。** 已核实 `renderer.js:98` `this.material = new MeshLambertMaterial(...)`，所有 chunk mesh 共用这**一个** `this.material`（`renderer.js:141`）。**若开新材质，draw call 从「≈区块数」裂成「区块数 × 材质数」，中端机性能灾难**——所以自定义贴图必须并入同一张图集、同一材质。
3. **迁移不传世界，靠人人持本地副本。** 资产也必须像 edits 一样「人人持本地副本」，新房主从本地副本重建资产清单并对后加入者重新分发。任何「只有原房主有 PNG」的设计在第一次迁移就丢图。

### 3.1 方案选定：内容寻址 + 分片 ws 传输 + 房主权威资产清单

| 方案 | 64KB 帧 | 无 CDN | 硬不变量 | 结论 |
|---|---|---|---|---|
| A. PNG base64 内联进 edit | **败**（64×64 PNG ≈ 4–8KB，叠 edits 爆帧；base64 +33%） | 通过 | 通过 | 否 |
| B. 中继加 HTTP 上传/下载路由 | 通过（绕开） | 勉强 | **败**（破坏哑中继零状态、只读、安全面） | 否 |
| **C. 内容寻址 + 分片 ws + 房主清单** | **通过（为它而生）** | **通过** | **通过（中继零改动）** | **选定** |

方案 C 把自定义贴图塞进引擎已证明能跑的同一范式：**世界 = 种子 + 增量 +（新增）资产清单**，三者都靠本地副本 + 房主权威 + 迁移收敛。

### 3.2 资产 = 内容寻址的小图集瓦片（不是任意大 PNG）

1. 玩家选 PNG → `createImageBitmap` 解码 → 画到**固定 32×32 离屏 canvas**（强制降采样）——这一步同时是显存安全阀和去重前提：无论传 4K 还是 16×16，落地永远 32×32 RGBA。
2. 取该 32×32 RGBA 字节做 SHA-256（`crypto.subtle.digest`），前 16 字节 hex = **资产 id（32 字符）**。内容相同 → id 相同 → 全房间天然去重。
3. 资产线格式 = 降采样后的 PNG 字节（`canvas.toBlob('image/png')`，32×32 像素艺术通常 0.5–2KB）。

> 自定义方块复用 `FACE_TILES` 结构——6 个面各指一个瓦片 id（内置 0..N 或自定义 `assetId`）。「做一个自定义方块」= 指定 6 个面分别用哪些已存在的瓦片资产。

### 3.3 单图集单材质纪律（吸收 minor #4，消除原文「分两张图集」的摇摆）

> **审查发现**：PNG 原文 §0 论证「必须单图集单材质」，§2.3 却又抛出「内置与自定义分两张图集」作为可选——若两张图集走两材质，正是自己否掉的方案；若共用一着色器，`MeshLambertMaterial` 单 `map` 不支持。同时原型—显现要把静态造物 tile 也烘焙进图集，与玩家自定义 tile 是否同一张无人对账。

**本文裁决：全部新瓦片（开发者静态造物 9..63 + 玩家自定义 64..255）共用同一张扩容图集 + 同一 `this.material`（NearestFilter 无 mip）。删除「分两张图集」选项。**

- 运行时单张 **512×512 RGBA 图集，16×16 = 256 槽**（512² RGBA 无 mipmap = **1MB 显存**，远低于事实基线 D2 的 256–512MB 预算）。
- 前若干槽留内置 + 已发布静态造物瓦片，其余给运行期自定义。
- `tileUV` 落地时扩展为可寻址这张 512² 图集（仍单材质单 map）。

### 3.4 资产清单 = 世界的第二种增量

```js
// 运行时（房主权威 + 每客户端本地副本）
assets:       Map<assetId, { bytes:Uint8Array, tile:int, refs:int }>   // tile 本地各自分配，不进线格式
customBlocks: Map<blockId(64..255), { faces:[assetId×6] }>             // 沿用 FACE_TILES 面序
```

**线格式（`joined` 扩展，只传清单不传字节）**：

```js
{
  ...,  // 现有 joined 字段
  rh: <REGISTRY_HASH>,
  customBlocks: [[blockId, [a0,a1,a2,a3,a4,a5]], ...],   // assetId 字符串引用
  assetIds:     ["<32hex>", ...]                          // 本世界用到的全部资产哈希（不含字节！）
}
```

**关键：`joined` 只传清单（哈希列表），字节单独按需分片拉取。** 一条 `assetId` ≈ 34B；`joined` 帧大小与「资产数量」线性、与「资产字节总量」无关，彻底回避 64KB 墙。**字节总量见 §4.2 的合帧预算（与 edits 同帧的总和必须 <64KB）。**

### 3.5 按需分片拉取协议（走现有 `msg`/`cast`，中继零改动）

```js
{ t:'asset-want',  ids:["<hash>",...] }                         // 成员→房主：我缺这些（收 joined 清单后比对本地）
{ t:'asset-chunk', id, seq, total, b64:"<≤48KB base64>" }       // 房主→某成员：分片（unicast）
{ t:'asset-new',   blockId, faces:[a0..a5], newIds:["<hash>",...] } // 房主→全体：有人新建自定义方块
{ t:'asset-put',   id, seq, total, b64, faces?, blockId? }      // 成员→房主：上传新资产（房主校验后落库）
```

- 每片 base64 ≤48KB（64KB 帧 - 信封 - 余量）。32² PNG 通常 1 片；最坏 16KB PNG base64 后 ~21KB 仍 1 片。分片机制主要为「未来放宽更大瓦片」和「批量清单」留安全协议。
- **房主侧重组校验铁律**：收齐 `total` 片 → 拼接 → `toBlob` 重算 SHA-256 → **必须等于 `id`，否则整个资产丢弃**（内容寻址自带防篡改，等同 `host.js:108-112` 对 block 的校验）。
- **限流**：房主对每个成员的 `asset-want` 按**每 rAF 帧最多发 2 片**（对齐「≤2 区块生成/≤2 网格重建/帧」的预算纪律）。后台标签无 rAF → 自然暂停，前台恢复继续。

### 3.6 「贴图开关=关」时如何完全跳过下载与常驻（显存安全阀，决定 #7）

**关 = 三件事一次全断：**

1. **不发 `asset-want`**：成员收到 `joined.assetIds` 后若本地「贴图=关」，直接跳过比对、一个字节都不拉。
2. **不解码、不上图集**：自定义方块的面全部回退到一个内置纯色瓦片（渲染时若 `assetId` 未解析，`tileUV` 指向「自定义缺省块」纯色 tile）。**自定义图集纹理根本不创建，显存可变部分归零**——这正是事实基线 D2 的结论：单材质单图集架构下，关掉自定义 PNG 即把可变显存归零。
3. **几何不变、玩法不变**：方块还在、还能碰撞、还能拆建（edit 不依赖贴图），只是长成纯色块。关的人和开的人能同房共存：贴图只是显现层，原型（blockId + 碰撞 + 掉落）对双方一致。

> 自动定档（决定 #7「按机型自动定档，绝不开局塞配置页」）：`deviceMemory < 4` 或触屏低端档 → 自定义贴图默认关，可手动开；旗舰默认开。

### 3.7 资产如何在房主迁移后对后加入者仍可得

严格对齐「迁移不传世界、靠人人持本地副本」：

1. **每客户端持有它见过的资产字节本地副本**：成员收齐的字节存进 `vc-assets`（新 localStorage 键，内容寻址 → 跨房间跨世界天然共享缓存）+ 运行时 `assets` Map。
2. **清单进世界存档**：`putWorldSave` 扩展为 `{room, seed, edits, customBlocks, assetIds, t}`（清单只是哈希，几 KB）；字节单独存 `vc-assets`（内容寻址 LRU）。
3. **迁移时（`main.js:841`）**：新房主用本地 `world` 重建 `HostRoom`（已有），额外从本地 `vc-assets` + 存档清单重建 `assets`/`customBlocks` Map。新房主是「被提升的最老成员」，只要它当初开着贴图玩过，本地就有全套字节——这正是「人人持本地副本」的红利。
4. **新房主对后加入者重新分发**：完全复用迁移后正常 join 流程，无特例。
5. **降级兜底（诚实面对边界）**：若新房主当初「贴图=关」地玩（本地只有清单无字节），它能下发清单但发不出字节 → 后加入者拉不到的资产**回退纯色块**（§3.6 同一机制）。世界结构（方块/碰撞/掉落）完全无损。可选 v2 增强：房主缺字节时向「清单声称持有」的其他成员经房主中转回收。

> 一句话：**方块（原型）永不丢，因为它是 edit + customBlocks 清单（随本地副本 + 迁移存活）；贴图（显现）最坏退化为纯色，但永不阻塞游戏。** 与已锁定决定 #2「原型稳定、显现可变」在工程上完美自洽。

### 4.0 堆地板锚（吸收 minor #5，把 256MB 显式声明为被设备封顶反推的设计地板）

### 4.2 64KB 单帧的联合预算（吸收 blocker 协调缺口 minor #2）

> **审查发现的跨文档缺口**：贴图设计把 `customBlocks` + `assetIds`（256 资产约 8.5KB+）塞进与 edits 同一个 `joined`/`resync` 帧；预算表把可迁移 edits 上限定在 ~3000 条（已是对 64KB 的保守墙）。两份各自不越 64KB，但叠加在同一帧的总和无人联合核算——edits 逼近 3000 条再叠 8.5KB 资产清单，可能推过 64KB 触发 ws 1009。

**本文裁决：把资产清单纳入同一帧联合预算，并定拆帧规则。**

| 帧内容 | 上限 | 依据 |
|---|---|---|
| 可迁移 edits | **≤ 2800 条**（从 ~3000 下调一档，给资产清单留固定预留） | A4 三档实测最保守 3119 条向下取整、再扣资产清单预留 |
| 资产清单（`customBlocks` + `assetIds`） | **≤ 8.5KB**（256 资产满载） | §3.4 线格式估算 |
| 信封 + 玩家名册 + 未来字段 | 余量 | `joined` 87B / `resync` 25B |

**拆帧规则（§2.3.3 / §3.4 共用）**：`joined`/`resync` 合帧前做大小检查，**>60KB（留 4KB 安全垫）则拆成独立帧**：
```js
{ t:'resync',   edits:[...] }              // 方块增量帧
{ t:'ent-sync', ents:[...] }               // 实体快照帧（§1.4）
{ t:'asset-sync', customBlocks, assetIds } // 资产清单帧（§3.4）
```
各自幂等合并、顺序无关。使「edits 墙 + 实体快照 + 资产清单」之和恒 <64KB。

### 4.3 ⚠ 中端机默认档：软阴影**不**默认开（吸收 minor #4 数值可行性发现）

> **审查发现**：把中端机默认定为「中=R3 + PCFSoft 软阴影」过于激进。软阴影 = 每帧多一遍场景深度 pass，R3 下 49 区块几何要再画一遍（draw call 近翻倍 49→~98），叠 1024² 阴影图采样。同帧还要背 HostRoom 给 7 成员转发 + 每帧最多 2 次 remesh（单次 16384 体素×6 面探测，中端 JS 约 2–3ms，2 次即 4–6ms）。深度 pass + 2 次 remesh + 渲染 + 转发同帧叠加，30fps/33.3ms 余量偏紧——与「保住随时被迁移成房主也不崩」纪律存在张力。

**本文裁决：中端机默认从「中=R3+柔和软阴影」收紧为「中-基线：R3 + 光影=关（无阴影，即当前引擎现状，0 额外 pass）」。** 软阴影列为：
- 中端机「可手动开」的项；
- **自适应监测帧时 >33ms 时第一个被自动关闭的项**（优先于 R3→R2、像素比→1.0）。

即让地板设备默认不承担深度 pass，把软阴影留给实测有余量或用户显式开启的场景。

### 4.4 光影三档精确定义（关键：全程非光追）

> **明确声明：本游戏不做、也不可能做实时光追。** WebGL2 无光追 API（`WEBGL_ray_tracing` 不存在），three.js r184 WebGL 后端无硬件光追路径。「高档光影」是用栅格化技巧仿体积感。

| 光影档 | 技术构成 | 现有引擎对照 | 性能代价 |
|---|---|---|---|
| **关** | 仅逐面 AO 假明暗（顶点色 +y=1.0/±z=0.85/±x=0.72/−y=0.5）+ 单方向光，不投阴影 | = 当前引擎现状（无 shadowMap） | 0 额外开销 |
| **柔和** | AO + `DirectionalLight` 开 PCFSoft 软阴影（贴图 1024²/2048²，**只覆盖玩家周边 R×CHUNK 的正交视锥**，不是全世界） | 新增（当前无 castShadow） | 多一遍深度 pass；仅旗舰/有余量时 |
| **高** | 柔和全部 + 泛光（Bloom）+ 体积感雾（加厚现有 `Fog` 营造空气透视） | 现有 `Fog` 已存在（`renderer.js:87`），Bloom 为新增 EffectComposer | 1 全屏后处理 pass + 1~2 降采样模糊；仅旗舰 |

- **柔和档软阴影边界（性能护栏）**：阴影只渲染玩家周边 R×CHUNK 米的正交相机；全世界投影会让阴影相机视锥爆掉，中端机扛不住。
- **泛光退化路径**：若 r184 的 `EffectComposer`/`UnrealBloomPass` 未随内置 `public/lib` 落地（当前 lib 仅 `three.core.js`/`three.module.js`，**无 examples/jsm**，需后续把 post 模块本地化，**严禁 CDN**），则「高档」泛光自动退化为「加厚雾 + 顶点色对比拉伸」，不引入外网依赖。

### 4.6 轻量模式世界硬预算数值表

反推三道硬墙：① 64KB 单帧（超限 ws 1009 关闭，实测一帧 ~3000–6000 条增量）；② JS 堆 ≤256MB（§4.0）；③ 30fps=33.3ms/帧（房主额外背 HostRoom 转发）。

### 4.7 自定义贴图预算：以 PNG 文档为权威（吸收 blocker #2 数值可行性发现）

> （同此搁置，见 §3 顶注：运行期自定义贴图字节通道休眠，搁置期瓦片槽由代码层素材占用。）

> **审查发现的互斥数值**：PNG 文档钉死「32×32 瓦片、单张 512² 图集、256 槽 = 1MB、禁 64×64+」；画质文档原文却写「256×256 RGBA、中 ≤4 张、高 ≤8 张、2MB」。256² 是 32² 的 64 倍像素，直接违反反膨胀铁律；单位（瓦片 vs 整张）、数量（256 vs 8）、显存（1MB vs 2MB）全对不上。

**本文裁决：以 PNG 文档为唯一权威，画质文档引用它。** 即：
- 单瓦片分辨率**恒 32×32**（不分档；清晰度档区分的是「内置瓦片 16 vs 32px」与「是否下载自定义」，**不区分自定义单张分辨率**）。
- 自定义瓦片图集常驻：**关=0 / 中=半张 512² 图集 / 高=满张 512²=1MB**。
- 删除互斥的「256×256 ≤8 张 = 2MB」表述。

### 4.8 自动定档 UX（不开局塞配置页，可手动覆盖）

> （「是否下载自定义」一档同此搁置，见 §3 顶注：搁置期无运行期自定义字节可下载，该档对内置+静态造物图集仍生效。）

**开局零打扰**（决定 #7）。首帧渲染前一次性探测打分，落到低/中/高：

| 探测信号 | 来源（无外网、纯本地 API） | 用途 |
|---|---|---|
| 触屏 vs 桌面 | 现有 `isTouchDevice()` | 触屏封顶中档 R3；桌面可到高档 |
| 逻辑核数 | `navigator.hardwareConcurrency` | <4 → 偏低；≥8 → 允许高 |
| 设备内存 | `navigator.deviceMemory` | ≤4GB → 低；≥8GB → 中/高候选 |
| 像素比 | `devicePixelRatio` | 高 dpr + 弱核 → 压像素比上限 |
| **首 ~3 秒实测帧时**（自适应回落） | 进世界后 rAF 帧时滑动均值 | 实测 >33ms 持续超阈 → 静默降档（软阴影→关 → R3→R2 → 像素比→1.0） |

- **默认档矩阵**：触屏中端机 → 中档（R3 / 60fps / **光影=关** / 像素比≤1.5）；触屏旗舰 → 高档；桌面 → 高档候选。**绝不让中端机自动落高档。**
- **热降频回落**：§D3 中端机持续 5–15 分钟热降频，自适应发现稳态帧时恶化即自动回落一档并轻提示一次（「已为流畅自动降低画质」，可改回）。**软阴影是第一个被砍的项**（§4.3）。
- **手动覆盖**：暂停菜单一个「画质」项（不在开局路径），3 个大按钮（省电/标准/极致）+「自动」开关（默认开）。进阶逐项藏「高级」折叠区。中端机选「极致」弹一次非阻塞确认（「可能掉帧或发烫，确定?」）。「贴图:关」旁注「(纯色块，最省显存)」点明它是显存安全阀。

### 4.9 预算与双系统边界

上表全是轻量模式（浏览器房主）天花板。重型模式（云服务器权威、真 tick、落盘不靠迁移）**不受 64KB 迁移帧约束**，活体实体/世界增量上限按服务器资源另定，可比轻量高 1–2 个数量级。**纯模拟内核两端共用同一份**——内核本身不写死这些上限，上限是**喂入侧（轻量房主）施加的护栏**。

## 6. 开放问题与已知风险

### 6.1 需要用户拍板的开放问题

1. **轻量模式 ~2800 条可迁移增量天花板**：本设计把「烘焙成 id」推到最省字节，但大型建筑/批量铺设仍撞墙。是否接受「大世界=重型模式专属」，还是后续要做 edits 分片协议（当前无）？
2. **作弊房主本地得益无解**（§2.6 B）：举报机制是否要做？还是直接靠「私有、熟人 LAN、不公开」的社交约束兜底？
3. **`REGISTRY_HASH` 强一致**：所有客户端必须同版本，版本升级需全员同步。是否接受这条运维约束（私有场景合理）？
4. **自定义贴图 255 槽/192 自定义方块耗尽**：逼近上限需新增协议，本设计标记为已知天花板。第一切片不会碰到，但中期要不要预研？（同此搁置，见 §3 顶注：运行期自定义字节通道当前休眠，256 槽搁置期由代码层素材占用；耗尽问题在上传/AI 管线重启后才现实化。）
5. **高档泛光是否值得本地化 `examples/jsm` post 模块**（严禁 CDN）：还是「高档=加厚雾」就够？

### 6.2 已知风险（已在设计中标注，列此备查）

| 风险 | 所在 | 缓解 |
|---|---|---|
| `surfaceHeight` 非纯只读，会拉起区块 | §1.3 | 实体钳在房主加载半径内，杜绝拉新区块 |
| 实体态迁移保真度低于静态世界（瞬移瑕疵） | §2.3.3 | 可接受瑕疵，与 edits 死窗口丢失同构；重型模式根治 |
| 软阴影在中端机吃紧 | §4.3 | 中端机默认关，列为自适应回落第一砍项 |
| id 三处校验口径不一致 | §2.4 | 落地必须收敛为同一谓词 `isKnownBlockId`，否则重开 A3 缺口 |
| 多份设计向同一帧叠加载荷 | §4.2 | 联合预算 + >60KB 拆帧规则 |
| 中端机 WebView 堆封顶随负载浮动 | §4.0 | 256MB 设计地板已留足安全系数 |
| three.js r184 内置 lib 无 `examples/jsm` | §4.4 | 泛光退化为加厚雾，绝不引 CDN |

### 6.3 与现有硬不变量逐条自检（成文结论）

| 不变量 | 结论 |
|---|---|
| 哑中继零逻辑 | ✅ 不破坏（实体/资产都是不透明 `d`，中继零改动） |
| 64KB 单帧 | ✅ 兼容（§4.2 联合预算 + 拆帧；快照/清单独立小帧） |
| host.js 零定时器 | ✅ 不破坏（内核零定时器，时间靠前台 rAF，后台冻结） |
| 种子 + 增量 | ✅ 不破坏（前两段一字不改，实体/资产正交新通道） |
| id 校验 + 静默丢弃 | ✅ 兼容扩展（三处收敛同一谓词，§2.4 关口已钉死） |
| 迁移不传世界 | ✅ 同构扩展（实体/资产同样「房主态胜出 + 本地副本」） |
| 无外网纯 ESM | ✅ 不破坏（全 P2P over ws，零 CDN，新模块纯 ESM） |
| 中端机预算 | ✅ 不破坏（≤256MB 堆地板、≥30fps、≤16 实体、贴图开关显存归零） |
