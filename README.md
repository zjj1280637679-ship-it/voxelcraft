# VoxelCraft

在浏览器里运行的跨平台(电脑 / 安卓 / iPhone)体素沙盒联机游戏。
无构建步骤、无外部资源依赖:克隆下来 `npm start` 即玩,局域网内手机无需外网。
核心理念:**谁创建房间,谁的浏览器就是这个房间的服务器**——后端只是一个不懂游戏的"哑中继"。

## 快速开始

```bash
npm install
npm start
```

需要 Node.js(项目按 Node 24 开发)。Windows 下也可以直接双击 `start.cmd`,自动开服并打开游戏页面。
只想跑服务器、不想装依赖?可以用压缩编译版:`node dist/server.js`(见下文"仓库结构")。

启动后控制台会列出所有可访问地址:

- 电脑本机:`http://localhost:8080`
- 手机(连**同一个 Wi-Fi**):访问控制台里列出的局域网地址,例如 `http://192.168.1.5:8080`

> 默认端口 8080,用环境变量 `PORT` 修改:PowerShell 下 `$env:PORT=8081; npm start`,
> macOS / Linux 下 `PORT=8081 npm start`。
> Windows 第一次启动时防火墙若弹窗询问是否允许 Node.js 联网,
> 必须勾选"专用网络"并允许,否则手机连不上。

## 玩法

1. **先建人物**:起名字、从 8 种颜色里分别选上衣和裤子配色,带实时预览。
   可以创建多个人物、随时切换,外观所有玩家可见。
   人物只存在浏览器本地(localStorage),无需注册任何账号。
2. **进入大厅**:自动连接服务器,房间列表分上下两段——
   - **我的足迹**(置顶):你进过的房间(本地最多记 20 条);在线的显示人数,
     不在线的显示"不在线";**自己创建过的房间名显示为金色**;每条右侧有 **×** 按钮,
     只删除这条本地足迹记录,绝不会影响线上的房间本身。
   - **其他公开房间**:服务器上的公开房间(最新创建的在前,最多列 50 个),
     点"刷新"更新,点任意一行直接进入。
3. **找房建房合一**:输入房间名(支持中文等任意 Unicode,1–16 个字符)点"进入房间"——
   有这个房间就直接进;没有则提示"没有找到「××」",点"创建这个房间"一键建房
   (可勾选公开/私密;私密房不出现在列表里,只能凭房间名进入)。
4. **房间名留空 = 随机开新房**:直接点"进入房间",自动生成一个随机中文房名
   (如"迷雾森林42")并直接创建,不再多问一步(默认公开)。
5. 游戏内**右上角"退出"按钮**随时回大厅(左上角显示当前房间名和人数)。
6. **房主退出,世界不消失**:房主离开(主动退出或断线)时,中继自动把最早加入的
   成员提升为新房主,世界无缝继续;只有当最后一个人也离开、无人可继承时,房间才消失。
7. 同一房间(上限 8 人 = 房主 + 7 名成员)内:实时看到彼此(含外观配色)、共享地形、
   方块修改全员同步,后加入的人也能看到之前的所有改动。
8. 分享房间:把 `服务器地址/?room=房间名` 链接发给朋友,打开后房间名已自动填好。

### 服务器地址栏

大厅顶部的"服务器地址"输入框决定连接哪个中继服务器:

- **留空 = 本站**:连回提供这个页面的服务器(网页版通常就这样用)
- `主机:端口`(如 `192.168.1.5:8080`)→ 按普通 `ws://` 连接(局域网风格)
- 不带端口的裸域名(如 `xxx.trycloudflare.com`)→ 按加密 `wss://` 连接(隧道/云风格)
- 写明 `http://`、`https://`、`ws://`、`wss://` 前缀时按前缀来

填过的地址会被记住,下次打开自动填上。

## 安卓 App 与公网入口

- 打包 APK:`powershell -File tools/build-apk.ps1 -DefaultServer "你的公网地址"`
  → 产物 `public/VoxelCraft.apk`(约 4.5MB,手机访问 `服务器地址/VoxelCraft.apk`
  即可下载安装;构建需要 JDK 与 Android SDK 环境)
- App 菜单里的"服务器地址"随时可改,打包时烤入的只是默认值,
  所以**换服务器/换隧道不用重装 App**
- 临时公网入口(测试用):`cloudflared tunnel --url http://localhost:8080`
  → 得到一个 `https://xxx.trycloudflare.com` 临时域名(每次重启域名会变)。
  长期部署方案见 `docs/scenarios/` 中的部署场景。

## 仓库结构

| 位置 | 内容 |
|---|---|
| `docs/whitepaper.md` | 白皮书:项目理念与整体设计的完整阐述 |
| `docs/scenarios/` | 场景集:deployment(部署)、protocol(协议)、worlds(示例世界) |
| `public/` | 客户端源码:全部游戏逻辑都跑在浏览器里(纯 ES Modules,无框架无打包器) |
| `server/` | 中继服务器源码:`server/index.js`,静态文件托管 + 游戏无关的转发 |
| `tools/` | 工具:中继协议自测、房主/成员命令行机器人、APK 打包脚本 |
| `dist/` | 压缩编译版:`node dist/server.js` 即可独立运行;改代码后用 `npm run build:dist` 重新生成 |
| `DESIGN.md` | 架构契约:**改任何代码之前必读** |

## 技术结构 v2:人人可开服

- **谁点"创建房间",谁的浏览器就是这个房间的服务器**——权威房间逻辑在
  `public/js/host.js`,纯事件驱动(不依赖定时器,后台标签页也能正常当房主)
- `server/index.js` 只是"哑中继":房间目录 + 不透明转发,**完全不解析游戏数据**。
  把它部署到任何一台有公网地址的便宜服务器上,所有人就都能远程开房/进房
- **房主迁移**:房主掉线时中继提升最早加入的成员接任,新房主向全房间广播
  完整修改集(resync),世界无缝继续——能这么做是因为每个玩家本地都持有
  完整世界(种子 + 全部修改记录)
- 世界由种子在各端本地生成,网络上只传"方块修改"和"玩家位置"(约 12 Hz),流量极小
- 调试:浏览器控制台 `window.__vc`(world / player / hostRoom / mode / doPlace 等)
- 测试工具:`node tools/relay-test.js`(中继协议自测,自带独立端口的临时服务器,
  逐项 PASS/FAIL);`tools/host-bot.js` / `tools/member-bot.js`(模拟房主/成员)

## 当前限制

- 无存档:世界跟着房主迁移链走,所有人都离开后房间消失
- 私密房间只靠房间名挡人,公网长期部署建议自行加访问控制
- 手机当房主可行(游戏会尽量保持屏幕常亮),但锁屏会冻结房间——
  好在会自动迁移给其他在线成员;长期房主建议用电脑
- 无生物、无合成、无水;方块共 8 种——内容层留给后续重新设计
